# SFT 数据构造

## 是什么

SFT 数据构造是指为监督微调（Supervised Fine-Tuning）准备训练数据的过程，包括数据源选择、数据生成、质量校验、格式统一、样本组织等多个环节。SFT 数据的质量直接决定了微调模型的效果上限——在高质量数据上训练的模型，即使调参不完美，效果也往往优于在脏数据上精细调参的模型。

SFT 数据构造涉及以下关键环节：

- **数据生成**：使用强模型（如 GPT-4、Claude）生成初始问答对，或用已有数据集改造为目标格式。
- **质量校验**：多模型交叉验证、人工抽检、规则过滤，确保每条数据符合质量标准。
- **多样性控制**：训练分布必须覆盖测试分布，避免某些模式在训练中缺失。
- **风格统一**：所有样本的输出格式必须严格一致，防止模型学偏。
- **样本组织**：难易程度标注、课程学习排序、单轮/多轮比例控制。

## 为什么重要

SFT 数据构造是微调工程中"投入产出比最高"的环节：

- **数据质量 > 模型参数**：400 条高质量精标数据的效果往往优于 4000 条脏数据的全量微调。数据清洗和增强带来的收益远超调参。
- **数据决定能力边界**：SFT 赋予模型"能力"，而能力的上限由数据的质量和多样性决定。如果训练数据覆盖了某种任务模式，模型就能学会；如果没有覆盖，再调参也无效。
- **错误数据产生错误模型**：一条错误的标注数据可能让模型产生系统性错误倾向。特别是在 SFT 数据量有限（几百到几千条）时，每条数据的信噪比极高。
- **训练分布须覆盖测试分布**：这是 SFT 数据构造中最核心的原则。当测试集效果差而训练集好时，优先检查测试集中是否存在训练集未覆盖的数据模式，然后针对性补充 5-10 条样本即可见效。

## 如何使用

### 强模型生成 + 多模型校验

使用比目标模型更强的模型（如 GPT-4、Claude）生成 SFT 训练数据，再用多个模型对生成的 QA 对进行交叉验证/打标，筛选出高质量样本：

1. **种子数据设计**：设计高质量种子 prompt 集合，每个 prompt 对应一个任务场景。
2. **强模型生成**：用 GPT-4/Claude 为每个种子 prompt 生成多条候选回答。
3. **多模型校验**：用 2-3 个不同的模型（如 GPT-4、Claude、Gemini）对每条候选回答打分或判断其是否符合质量标准。
4. **一致性筛选**：保留多模型一致认为高质量的样本。如果多模型打分不一致，该样本可能处于边界，建议人工复核或丢弃。
5. **人工抽检**：有资源的情况下，至少抽检 10-20% 的样本，确认质量。

### 多样性要求

- **训练集分布必须覆盖测试集分布**：在收集数据前先定义目标测试场景的分布，确保每条训练数据对应的模式在测试集中都存在对应样本。
- **不遗漏关键模式**：如果测试集包含 API 调用、JSON 输出、多轮对话、代码生成等模式，训练集必须全部覆盖。
- **针对性补充**：当发现测试集效果差时，首先检查测试集的 bad case 在训练集中是否有相似样本。没有就手动添加 5-10 条。
- **每个垂域 ~400 条高质量数据**：对于多领域场景，每个垂直领域约 400 条高质量样本足以让模型学会该领域的回答范式。关键在"高质量"而非"多数量"。

### 输出风格统一

风格一致性是 SFT 数据构造中最容易被忽视但又最重要的环节之一：

- 如果输出要求 JSON 格式，所有样本必须全部使用 `\`\`\`json` 包裹，不能混入无包裹的纯 JSON。
- 如果 SQL 答案统一用 `select a, b, c` 格式，就永远不要出现 `select *`。
- 如果回答使用 Markdown 列表，所有样本的列表风格必须一致（用 `-` 就全用 `-`，用 `*` 就全用 `*`）。
- 风格一致性直接决定 SFT 后模型在实际使用中的输出稳定性。混入多种风格会让模型"学偏"——表现在训练 loss 正常，但模型输出格式混乱。

### 单轮与多轮比例控制

单轮对话与多轮对话数据的混合比例是 SFT 数据层面的重要超参数：

- **纯 QA 场景**（如问答机器人、分类）：单轮占比 80-100%。
- **对话助手场景**（如客服、聊天）：多轮占比 30-50%。
- 比例失衡会导致模型在某种对话模式下表现极差。
- 最佳比例取决于下游应用，建议作为超参数进行实验搜索。

### 课程学习：由易到难

对数据集标注难易程度，训练时先喂简单样本、再逐步增加困难样本的比重：

- **难度标注维度**：回答长度、逻辑复杂度、是否需要外部知识、是否需要多步推理。
- **训练策略**：前 20% 的 training steps 以简单样本为主，中间 60% 均匀混合，最后 20% 增加困难样本比例。
- **效果**：可以加速模型收敛，尤其在数据量大、任务多样时效果明显。

### SFT 不需要思维链和 Few-shot

SFT 训练数据中不需要构造 CoT 或 Few-shot 格式：

- CoT 和 Few-shot 是推理时激发模型能力的 prompt 技巧，不是训练格式。
- SFT 直接用大量高质量、风格统一的问答对样本让模型学习，效果更好且标注成本更低。
- 但需注意：在数学/逻辑推理等需要多步推理的任务中，CoT 数据的价值可能更大，上述建议取决于任务类型。

## 常见误区

### "SFT 数据越多越好"

数据量不是越多越好。SFT 不同于预训练——数据量通常只有几千到几万条。超过某一阈值后，增加的样本不再带来明显收益，甚至可能引入噪声。关键在于每一条数据的质量。

### "多模型校验可以完全取代人工校验"

多模型校验可以快速过滤明显低质的样本，但模型可能在某些维度上存在共同的偏差（如偏好更长或格式更花哨的回答）。有人工资源时建议至少做 10-20% 的抽检。

### "数据风格统一是小事"

输出风格不一致是 SFT 中最隐蔽的 bug 之一。训练 loss 看起来正常，模型也能回答问题，但输出格式五花八门——有时是纯文本，有时是 JSON，有时带 Markdown。这种问题排查极其困难，因为"第一个出现问题的 checkpoint 往往在中后期才发现"。

### "课程学习对所有任务都有效"

课程学习在数据量大、任务多样时效果显著，但对于数据量极少的场景（如<200 条）或者任务高度同质的场景，难度排序带来的收益有限。简单场景不需要课程学习。

### "SFT 数据可以一次性构建完成"

SFT 数据构建是一个迭代过程。先构建最小可用数据集（~200 条/领域）微调模型并评测，根据评测结果中的 bad case 补充数据，重复此过程 2-3 轮。一次构建往往无法覆盖所有关键模式。
