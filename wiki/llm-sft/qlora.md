# QLoRA 量化低秩适配

## 是什么

QLoRA（Quantized Low-Rank Adaptation）是在 LoRA 基础上引入量化技术，将基座模型权重量化到 4-bit，同时保留 LoRA 适配器以 BF16/FP16 精度训练，从而在单卡消费级 GPU 上实现大模型微调。

QLoRA 和 LoRA 的核心区别在于基座模型的存储和计算精度：

- **LoRA**：基座模型以 BF16/FP16 加载，前向传播和反向传播都在 BF16/FP16 下完成。
- **QLoRA**：基座模型以 NF4 量化格式存储，前向传播时反量化到 BF16/FP16 参与计算，反向传播时只计算 LoRA 适配器的梯度，基座权重不更新。

QLoRA 的核心创新包括三项技术：

- **NF4 (Normal Float 4)**：一种为神经网络权重分布定制化的 4-bit 量化数据类型。与均匀量化的 FP4 不同，NF4 利用正态分布的分位点来分配量化区间，使得量化误差在权重分布密集区域（均值附近）最小化。NF4 的精度损失显著低于 FP4，这是 QLoRA 能够在极低精度下保持效果的关键。
- **双重量化 (Double Quantization)**：对量化缩放因子（scale factors）本身再进行一次量化。标准 4-bit 量化需要为每 64 或 128 个参数存储一个 FP32 的 scale factor，这会产生额外的显存开销。双重量化将这些 scale factor 量化到 8-bit，节省约 0.5 bits/parameter 的额外开销。
- **Paged Optimizers**：利用 CPU 和 GPU 之间的统一内存管理，当 GPU 显存不足时将优化器状态分页到 CPU 内存，训练过程中按需调回。这进一步降低了峰值显存需求，使得在 24GB 显存上微调 33B 模型成为可能。

训练时，基座模型权重保持 NF4 量化状态，不计算梯度。LoRA 适配器 (A, B) 以 BF16/FP16 精度训练。前向传播时，量化权重被反量化到计算精度参与计算。

## 为什么重要

QLoRA 将大模型微调的门槛从"至少需要 48GB 显存（LoRA 微调 7B）"降到"12GB 显存可微调 7B 模型，24GB 可微调 33B 模型"。

具体记忆体占用对比（7B 模型）：

| 方法 | 基座精度 | 可训练参数 | 预估显存 |
|------|---------|-----------|---------|
| 全量微调 | BF16 | 100% | ~112GB |
| LoRA | BF16 | ~0.1% | ~16-20GB |
| QLoRA | NF4 | ~0.1% | ~10-12GB |

QLoRA 使得在 RTX 3090/4090（24GB）上微调 33B 模型、甚至是 65B 模型成为可能，极大降低了大模型微调的硬件门槛。但需要明确：QLoRA 的代价是训练速度的降低（反复量化和反量化增加计算开销）和效果的轻微下降（量化噪声）。

### 训练稳定性考量

QLoRA 相较 LoRA 引入额外的训练不稳定因素：

- **量化-反量化往复带来的误差累积**：每次前向传播都经过量化→反量化，这个过程中损失的精度无法恢复，且随训练步数累积。
- **NF4 的低精度限制**：虽然 NF4 针对正态分布做了优化，但在权重分布极不均匀的层（如 embedding 层）量化损失更大。
- **梯度流经量化层**：梯度需要回传到量化权重上，量化导致的梯度噪声会影响 LoRA 适配器的更新质量。
- **与混合精度交互**：反量化后的计算精度（BF16/FP16）与量化精度（4-bit）之间有巨大的精度差距，某些操作可能放大这个差距。

稳定性对策：使用 BF16 而非 FP16、降低 LR 从 1e-4 起步更保守、开启梯度检查点、监控训练和验证 loss 的差距（差距异常大可能说明量化噪声过大）。

## 如何使用

### NF4 vs FP4 的选择

NF4 是 QLoRA 的核心贡献，应作为默认选择：

- **NF4**：基于权重正态分布的分位点量化，在均值和常见值附近的量化误差最小。QLoRA 论文实证 NF4 优于 FP4。
- **FP4**：均匀量化，在权重分布均匀时表现尚可，但神经网络权重通常呈正态分布，FP4 在均值附近的量化精度不如 NF4。
- 除非有特殊兼容性需求，始终优先使用 NF4。

### 双重量化的开关

双重量化节省约 0.5 bits/parameter，对 7B 模型约节省 0.4GB，对 65B 模型约节省 3.5GB。在显存极度受限时应启用。如果显存充裕可以关闭，以减少一点点反量化计算开销。

### 计算精度选择

QLoRA 训练的计算精度是关键调参维度：

- **BF16 优先**：只要硬件支持（Ampere 及以上架构，即 RTX 30xx/A100/H100），始终使用 BF16。BF16 的指数位与 FP32 相同（8 bits），能表达更大的数值范围，不会出现 FP16 常见的溢出问题（overflow/underflow）。QLoRA 训练中，反量化后的计算通常在 BF16 下更稳定。
- **FP16**：在无 BF16 硬件支持时使用（如 GTX 10xx/V100）。需注意：FP16 的精度范围窄，可能出现 loss 溢出不降、梯度爆炸等问题。开启 loss scaling 是必须的。
- **不推荐 FP32**：QLoRA 的主要优势就是显存节省，使用 FP32 训练会浪费这一优势。

### 梯度检查点 (Gradient Checkpointing)

QLoRA 场景下应默认开启梯度检查点。它可以显著降低激活值占用的显存（约 50-70%），代价是增加约 20-30% 的训练时间。由于 QLoRA 本身训练速度较慢，显存与速度的折中需要根据实际硬件情况取舍。

### Sequence Packing

Sequence Packing 在 QLoRA 中尤其有用——将多个短样本拼接到一个训练序列中，提高 GPU 利用率。打包时需注意：

- Attention mask 必须正确设置，确保不同样本之间不互相"看到"。
- Position ID 在每个样本内应重新从 0 开始。
- 打包策略的选择（按长度相似度打包 vs 随机打包）会影响训练稳定性。

### QLoRA 的 LR 策略

QLoRA 的学习率应比 LoRA 略微保守：

- 从 1e-4 起步（低于 LoRA 的 2e-4 起点），因为量化噪声使梯度信号质量下降，大 LR 更容易震荡。
- 如果训练 loss 稳定且收敛良好，可以逐步增大到 2e-4。
- 监控验证集指标来确认 LR 是否合适，不要只看训练 loss。

## 常见误区

### "QLoRA 效果等同于 LoRA"

QLoRA 因为引入了量化噪声，训练效果略低于 LoRA。QLoRA 论文报告 NF4+Double Quantization 在大多数任务上达到 LoRA 的 95-99% 效果。量化本身引入了信息损失，且训练时反复量化和反量化会累积误差。如果显存充裕，优先使用 LoRA（BF16）而非 QLoRA。

### "QLoRA 不需要调参"

QLoRA 的调参维度比 LoRA 更多：量化配置（NF4 vs FP4）、计算精度（BF16 vs FP16）、是否开启双重量化、是否启用梯度检查点、Sequence Packing 策略——每一个都会影响训练稳定性和最终效果。

### "BF16 和 FP16 可以互换"

不可以在代码中直接把 dtype 从 bf16 改为 fp16 而不做其他调整。FP16 需要 loss scaling、更小的 LR，且对梯度溢出更敏感。BF16 不稳定时通常是 LR 或其他超参的问题，而非精度不够。

### "QLoRA 可以无脑拉长上下文"

长上下文场景下 QLoRA 需要特别处理：显存随序列长度平方增长（attention 部分），NF4 量化带来的显存节省可能被长序列的 attention 计算消耗抵消。同时位置编码的外推能力有限——不是直接把 `max_length` 改到 32k 就能工作。

### "量化模型不适合做敏感任务"

量化确实引入了噪声，但 NF4 针对权重分布特性设计，对于大部分文本生成任务，量化带来的精度损失在可接受范围内。如果任务对生成质量极其敏感（如医疗建议、法律文书），建议使用 BF16 LoRA 而非 QLoRA。

### "QLoRA 训练速度和 LoRA 一样快"

QLoRA 由于每次前向传播都需要执行反量化操作（NF4 -> BF16），训练速度比 LoRA 慢约 20-40%。如果速度是优先考虑因素且显存充足，应选择 LoRA。
