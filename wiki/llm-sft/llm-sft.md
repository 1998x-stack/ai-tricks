# 大模型SFT

## 概述

SFT（Supervised Fine-Tuning）是在预训练大模型基础上，使用高质量标注数据对模型进行有监督微调，使模型学会遵循指令、适应特定输出格式和领域风格的过程。SFT 赋予模型"能力"，而 RL 将这个能力推向"上限"——如果基模本身都无法做出 20% 的正确回答，SFT 成功的概率极低。

## 核心技巧

### Prompt Engineering First（先调提示词，再做SFT）
- **来源**: Tang AI
- **适用场景**: 任何准备做 SFT 的任务
- **原文**: "先调prompt，保证让模型可以按照你想输出的样式先输出。如果怎么调都不行建议换模型，说明能力不行。正常来说，通过prompt至少有20%以上的答案是正确的，才有SFT成功的可能性。"
- **要点**: 在做任何微调之前，必须先用 prompt 验证基座模型是否具备完成任务的基本能力。如果 prompt 调优后正确率达不到 20%，说明模型本身能力不足，应该换更强的基座模型，而不是尝试用 SFT 来"补课"。如果 prompt 不复杂就能有不错效果，也不用把 prompt 构造得太复杂。
- **参见**: [[../llm-rl/llm-rl]]

### 数据构造：强模型生成 + 多模型校验
- **来源**: Tang AI
- **适用场景**: SFT 训练数据收集与质量保证
- **原文**: "可以通过更强大的模型造一些数据，然后，再多个模型对其进行打标，判断产生的QA对是否符合要求，这样够可以构造一批高质量的样本。如果有人力的支持，可以进行人工检验，更加保险。"
- **要点**: 使用比目标模型更强的模型（如 GPT-4、Claude）来生成 SFT 训练数据，再用多个模型对生成的 QA 对进行交叉验证/打标，筛选出高质量的样本。有人工校验资源的情况下，加一道人工检验最保险。
- **参见**: [[../data-eval/data-eval]]

### 学习率与权重衰减范围
- **来源**: Tang AI
- **适用场景**: SFT 超参数初始化
- **原文**: "调参的时候，可以对learning rate，weight decay从大往小调，通常lr从1E-4开始，weight decay从0.25开始。"
- **要点**: SFT 的 LR 从 1e-4 开始尝试，从大到小搜索；weight decay 从 0.25 开始，也是从大到小调。Tang AI 的经验是这两个参数从大往小调比从小往大更容易找到合适值。
- **参见**: [[../optimizer-lr/optimizer-lr]]

### 数据多样性：训练分布须覆盖测试分布
- **来源**: Tang AI
- **适用场景**: 当测试集效果不佳时排错
- **原文**: "数据的多样性非常重要，再三强调，训练集的数据分布一定要包括测试集的数据分布，这样测试集才会有好的效果，所以，当你的测试集效果不好，但是训练集数据还可以的时候，记得去检查测试集有没有类似的数据。如果没有，ok，手动人工针对性添加数据把，先5个，10个的先加，再看训练效果。"
- **要点**: 这是 SFT 数据构建中最核心的原则之一。当发现测试集效果远不如训练集时，优先检查测试集中是否存在训练集未覆盖的数据模式和分布，然后针对性地补充少量数据（先加 5-10 条）即可见效。
- **参见**: [[../data-eval/data-eval]]

### 输出风格统一性
- **来源**: Tang AI
- **适用场景**: SFT 数据格式设计
- **原文**: "数据集的风格一定要统一，输出的答案风格一定要单一。比如，每次答案都是以json格式的，确保数据都是\`\`\`json xxx \`\`\`这种格式。如果sql的答案都是select a,b,c, 那么就不要出现select \*。保证你数据的统一性，纯洁性是模型不学偏的根本。"
- **要点**: SFT 数据中所有样本的输出格式必须严格统一。混入多种风格会让模型"学偏"——例如，JSON 格式就全部用 \`\`\`json\`\`\` 包裹，SQL 答案就全部统一列名。输出风格的一致性直接决定了 SFT 后模型在实际使用中的稳定性。

### SFT 不需要思维链和 Few-shot
- **来源**: Tang AI
- **适用场景**: SFT 数据策略
- **原文**: "思维链+fewshot肯定是可以提高原始模型能力的，但是，SFT就不用加思维链和fewshot了，直接用样本堆死它。而且思维链+fewshot也超级耗时啊，标注也是一个超级麻烦的事情。所以，还不如直接造大量高质量样本，直接让模型学习。暴力但是很有效。"
- **要点**: CoT 和 Few-shot 是 prompt 层面的技巧，用于在推理时激发模型能力。但在 SFT 训练中，不需要在每条数据中构造 CoT 或 Few-shot 格式——直接堆大量高质量的、风格统一的问答对样本，让模型直接从数据中学习，效果更好且标注成本更低。
- **参见**: [[../llm-rl/llm-rl]]

### 每领域约400条高质量数据即可
- **来源**: Tang AI
- **适用场景**: 多任务/多领域 SFT 数据量决策
- **原文**: "对于多任务问题，可能有的同学会问到底多少条数据才合理，个人经验，每个垂域400条高质量数据足矣。"
- **要点**: 对于每个垂直领域，大约 400 条高质量标注数据就足够让模型学会该领域的回答范式。关键在"高质量"——宁可 400 条精标，不要 4000 条脏数据。
- **参见**: [[../data-eval/data-eval]]

### 课程学习：由易到难
- **来源**: Tang AI
- **适用场景**: SFT 训练策略
- **原文**: "可以对数据集设置一个难易程度，让模型先学简单后学难的，有利于模型的加速收敛。"
- **要点**: 在 SFT 数据中标注难易程度，训练时先喂简单样本、再逐步增加困难样本的比重，可以加速模型收敛。这种方法在数据量大、任务多样时效果尤其明显。
- **参见**: [[../optimizer-lr/optimizer-lr]]

### Loss 不可靠，必须看 Checkpoint 指标
- **来源**: Tang AI
- **适用场景**: SFT 训练监控
- **原文**: "事实上loss，对于大模型训练仅能做一个基本的判断了，基本上，到了中后期，loss小，效果不一定好，甚至还会出现loss先下降后上升的情况，即使是在训练集。所以，正确的判断方法还是，从小步数的checkpoint，进行accuray或者其他业务指标的判断，正常情况，只要模型在正常训练，该指标通常也是下降逐渐到收敛的一个状态。因此，不能太依赖loss指标，仅仅做初步参考。"
- **要点**: 大模型 SFT 训练中后期，loss 与模型实际效果之间关联度很低。可能出现 loss 下降但效果变差，或者 loss 先降后升的情况。正确做法是定期保存 checkpoint，在每个 checkpoint 上跑业务指标（准确率、F1、人工评测等）来评估效果。Loss 仅作为训练是否正常的初步参考。
- **参见**: [[../data-eval/data-eval]]

### 模型越大效果越好
- **来源**: Tang AI
- **适用场景**: 基座模型选型
- **原文**: "模型越大，肯定是效果越好，在模型很大情况下可以开ds。"
- **要点**: 基座模型参数量越大，SFT 后效果通常越好。在超大模型训练时，应开启 DeepSpeed（ZeRO 等）来管理显存和训练效率。

### LoRA 调参指南
- **来源**: BugBuster喵
- **适用场景**: LoRA 微调的超参数选择
- **原文**: "LoRA 调参里，最常调的是学习率、rank、alpha、dropout、target modules、训练轮数、最大长度、batch。rank 常见从 8、16、32 开始，很多业务任务 rank 8 或 16 已经够用。rank 不是越大越好，数据少时大 rank 更容易过拟合。alpha 常见 16、32、64。LoRA dropout 0 到 0.1 之间试。学习率 1e-4 到 3e-4 是常见范围，但小数据、强基座模型可以更低。训练轮数不要太多，很多 SFT 两三个 epoch 已经开始记数据了。"
- **要点**: LoRA 调参要点：
  - **Rank**: 8/16/32 起步，多数业务任务 8 或 16 足够。数据量少时大 rank 反而容易过拟合。
  - **Alpha**: 16/32/64 常见。
  - **Dropout**: 0~0.1 范围内尝试。
  - **LR**: 1e-4 到 3e-4 是常见范围，小数据或强基座模型可以用更低。
  - **训练轮数**: 不要太多，2-3 个 epoch 后模型可能已经开始"记数据"而非学习范式。
- **参见**: [[../optimizer-lr/optimizer-lr]]

### QLoRA 的代价与稳定性
- **来源**: BugBuster喵
- **适用场景**: 显存受限时的 QLoRA 微调
- **原文**: "QLoRA 很省显存，但别以为它没有代价。量化配置、计算精度、梯度 checkpoint、sequence packing 都会影响训练稳定性。bf16 能用就用。长上下文训练要特别注意显存和位置编码适配，不是把 max length 改到 32k 就完事。"
- **要点**: QLoRA 虽然大幅降低了显存需求，但引入了额外的调参维度：量化配置（4-bit NF4 vs FP4）、计算精度、是否开启梯度 checkpoint、是否使用 sequence packing，这些都会影响训练稳定性和最终效果。优先使用 bf16（硬件支持前提下），长上下文训练时要特别注意位置编码的适配——不是简单改 `max_length` 就能解决。
- **参见**: [[../infra/infra]]

### Chat Template 一致性
- **来源**: BugBuster喵
- **适用场景**: SFT 训练与推理的配置对齐
- **原文**: "大模型微调还有一个常见坑：chat template 不一致。训练用一种模板，推理用另一种模板，效果会非常难看，而且不容易第一时间发现。"
- **要点**: 训练时使用的 chat template 必须与推理时完全一致，否则模型会出现严重的格式混乱和回答退化。这个 bug 极其隐蔽，因为训练 loss 看起来正常，但模型输出一塌糊涂。建议将 template 写入训练配置并通过单元测试验证训练和推理的一致性。
- **参见**: [[../infra/infra]]

### SFT 数据混合比例
- **来源**: BugBuster喵
- **适用场景**: SFT 数据集构成
- **原文**: "大模型 SFT 里，单轮和多轮数据比例就是超参数。"
- **要点**: 单轮对话数据与多轮对话数据的混合比例是 SFT 数据层面的重要超参数。比例失衡会导致模型在某种对话模式下表现差。具体最佳比例取决于下游应用场景——纯 QA 场景单轮占比高，对话助手场景多轮占比高。建议作为超参数进行实验搜索。
- **参见**: [[../data-eval/data-eval]]

### Packing 策略
- **来源**: BugBuster喵
- **适用场景**: SFT 训练效率优化
- **原文**: "大模型微调里，数据调参包括...是否 packing、最大长度设置..."（上下文多处提及 sequence packing）
- **要点**: Sequence Packing 将多个短样本拼接为一个训练序列，可显著提升训练吞吐和 GPU 利用率。但 packing 需要正确处理 attention mask 和 position ID，确保不同样本之间不会互相干扰。packing 策略的选择（按长度相似度打包 vs 随机打包）也会影响训练稳定性。
- **参见**: [[../infra/infra]]

### Max Length 设置
- **来源**: BugBuster喵
- **适用场景**: SFT 训练数据截断策略
- **原文**: "长上下文训练要特别注意显存和位置编码适配，不是把 max length 改到 32k 就完事。"
- **要点**: `max_length` 不是可以无脑设大的参数。增加 max length 会：
  - 大幅增加显存消耗（attention 计算量随长度平方增长）
  - 需要位置编码适配（RoPE 等的外推能力有限）
  - 可能导致训练不稳定
  建议根据实际数据分布设定，统计训练数据中 95% 分位的长度作为 max length 起点，再进行微调。
- **参见**: [[../infra/infra]]

### SFT 常见陷阱
- **来源**: BugBuster喵
- **适用场景**: SFT 训练流程自查
- **原文**: "大模型 SFT 里，assistant 部分和 user 部分都参与 loss，模型学得乱七八糟。"
- **要点**: SFT 训练中几个最常见的"调参之前先修 bug"场景：
  1. **Loss 计算包含 user 部分**: SFT 时 loss 应只计算 assistant 回复部分的 token，user 问句和 system prompt 应该被 mask 掉。如果 user 部分也被计入 loss，模型会学到"预测用户的问法"，严重干扰训练效果。
  2. **Tokenizer 不一致**: 训练时使用的 tokenizer 与推理时不匹配，导致生成结果异常。
  3. **训练和推理预处理不一致**: 数据清洗、截断策略、特殊 token 处理等必须完全相同。
  4. **Chat template 不一致**（见上条独立技巧）。
- **参见**: [[../data-eval/data-eval]]

### 什么时候不应该用 SFT（改用 RAG）
- **来源**: BugBuster喵
- **适用场景**: 技术方案选型
- **原文**: "别把微调当成万能开关。很多企业知识类问题，知识应该放在检索系统里，不该硬塞进模型参数里。微调用来调风格、格式、流程、领域表达，检索用来提供事实依据。这个判断做对，比调 50 组参数更重要。"
- **要点**: SFT 适合用来调整模型的输出风格、格式规范、领域表达方式和流程遵循能力，不适合用来注入知识。当任务涉及大量企业私有知识、实时信息、长文档依据时，应优先使用 RAG（检索增强生成），而不是尝试通过 SFT 将知识"硬编码"到模型参数中。RAG 比 SFT 更灵活、成本更低、更新更及时。
- **参见**: [[../llm-rl/llm-rl]]

### SFT vs RL 的关系
- **来源**: Tang AI、BugBuster喵
- **适用场景**: 整体技术路线理解
- **原文**: "SFT是为了让模型有能力去回答问题，而RL仅仅是将这个能力推到上限。"
- **要点**: SFT 和 RL 不是二选一的关系，而是递进关系：
  - **SFT 给能力**: 让模型学会遵循指令、输出特定格式和风格。
  - **RL 推上限**: 在 SFT 的基础上，让模型在偏好维度上做到更好。
  - 如果 SFT 效果差，RL 基本不可能好（"如果运动员只能 10 中 2，他怎么都到不了 10 中 8"）。
  - 对于有确定性 rule reward 的任务（如数学、逻辑），可以跳过 SFT 直接做纯 RL；但反之——没有 SFT 先给能力，RL 难以奏效。
- **参见**: [[../llm-rl/llm-rl]]

### SFT 训练轮数不宜过多
- **来源**: BugBuster喵
- **适用场景**: SFT 训练停止条件
- **原文**: "训练轮数不要太多，很多 SFT 两三个 epoch 已经开始记数据了。"
- **要点**: SFT 不同于预训练，数据量通常有限，几轮 epoch 后模型就会从"学习范式"转向"记忆数据"。建议在验证集上监控指标变化，一旦指标不再提升或开始下降就停止。通常 2-3 个 epoch 是常见的合理范围。
- **参见**: [[../data-eval/data-eval]]

### 全量微调的学习率范围
- **来源**: BugBuster喵
- **适用场景**: 预训练模型全量参数微调
- **原文**: "预训练模型全量微调，尤其是 BERT、RoBERTa、DeBERTa、ViT 这类，常见范围在 1e-5 到 5e-5，别上来就 1e-3。"
- **要点**: 全量微调（Full Fine-tuning）的学习率通常比 LoRA 低一个数量级，在 1e-5~5e-5 范围内。这与 LoRA 的 1e-4~3e-4 不同，因为全量微调同时更新所有参数，需要更小的步长来保护预训练权重不被破坏。
- **参见**: [[../optimizer-lr/optimizer-lr]]

### SFT 数据的"难易程度"标注
- **来源**: Tang AI、BugBuster喵
- **适用场景**: SFT 数据组织和训练调度
- **原文**: "可以对数据集设置一个难易程度，让模型先学简单后学难的" + "大模型微调里，数据调参包括指令格式、答案风格、样本去重、低质量样本过滤"
- **要点**: 除了课程学习（由易到难）外，SFT 数据管理还包括：样本去重、低质量样本过滤、拒答样本比例控制、领域样本与通用样本的混合比例等。这些数据层面的调参往往比模型参数调优更值钱。
- **参见**: [[../data-eval/data-eval]]

## 总结

SFT 的成功高度依赖三个核心要素：

1. **基座模型选对**: prompt 层面先验证模型有 20%+ 的基本能力，否则换模型。
2. **数据质量优先**: 风格统一、分布覆盖、~400 条/领域高质量数据 > 海量脏数据。数据构造使用强模型生成 + 多模型校验。
3. **参数从简到繁**: 先用保守参数跑通，验证训练流程无误（特别是 loss mask、chat template 等陷阱），再对 LR、weight decay、LoRA rank/alpha 等参数进行系统性搜索。不要迷信 loss 曲线，以 checkpoint 上的业务指标为准。
