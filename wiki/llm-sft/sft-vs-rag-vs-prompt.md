# SFT vs RAG vs Prompt Engineering：技术方案选型

## 是什么

在大模型应用开发中，面临四种主要技术手段来改善模型输出：Prompt Engineering（提示词工程）、RAG（检索增强生成）、SFT（监督微调）、RL（强化学习对齐）。它们各自解决不同维度的需求，选择的正确性比具体调参技巧更重要。

四种手段的核心定位：

- **Prompt Engineering**：通过精心设计的指令和示例，在不修改模型参数的情况下引导模型行为。成本最低、迭代最快。
- **RAG**：将外部知识库作为推理时的检索依据，模型从检索结果中提取信息回答问题。用于注入实时信息和私有知识。
- **SFT**：通过高质量标注数据微调模型，让模型学会特定的输出风格、格式规范和领域表达方式。
- **RL**：在 SFT 基础上通过偏好学习使模型"推向上限"，优化模型在偏好维度上的表现。

## 为什么重要

技术方案选型的错误是成本最高的错误——它决定了后续投入的方向：

- **选错方案浪费资源**：如果一个任务通过 prompt engineering 就能解决，做 SFT 是数周的数据收集和训练资源的浪费。同样，如果任务依赖大量企业私有知识，做 SFT 来"硬编码"知识效果不如 RAG 的 10%。
- **SFT 不是万能开关**：SFT 擅长调整输出风格、格式和表达方式，但不擅长注入知识。把知识塞进模型参数会让模型在数据量不足时产生幻觉，且知识更新需要重新训练。
- **RAG 比 SFT 更适合知识密集型任务**：RAG 的灵活性和成本优势明显——知识更新只需替换检索库，无需重新训练模型；检索可以精确控制信息来源；对比 SFT 需要大量高质量标注数据，RAG 的成本极低。
- **Prompt Engineering 是前置验证步骤**：在做任何微调之前，必须先用 prompt 验证基座模型是否具备完成任务的基本能力。这是决策链中的第一步。

## 如何使用

### 20% 规则：Prompt 前置验证

在做任何微调决策前，先用 prompt 测试基座模型的任务能力：

1. **设计一组 10-20 条代表性 prompt**：覆盖目标任务的主要模式和难度层次。
2. **评估基座模型的原始输出**：看模型在不经任何调优的情况下能正确回答多少比例。
3. **优化 prompt 后再次评估**：用 system prompt、few-shot 示例、输出格式约束等技巧尝试提升正确率。
4. **应用 20% 阈值**：
   - 如果优化 prompt 后正确率 < 20%：模型本身能力不足，换更强的基座模型。不要尝试用 SFT 来"补课"——基模做不到的，SFT 也做不到。
   - 如果正确率在 20-60%：说明模型具备基本能力，SFT 可以显著提升效果。这是 SFT 的最佳目标场景。
   - 如果正确率 > 60%（且不需要特定输出格式）：prompt engineering 已足够，不需要 SFT。继续优化 prompt 即可。

### 决策框架

根据任务特征选择最合适的技术方案：

| 任务特征 | 推荐方案 | 理由 |
|---------|---------|------|
| 需要特定输出格式（JSON、表格、代码风格） | SFT | 格式学习是 SFT 最擅长的 |
| 需要遵循特定流程或多步规则 | SFT | 流程遵循能力需要训练数据教会模型 |
| 需要领域风格和表达方式 | SFT | 风格统一性是 SFT 数据的核心要求 |
| 涉及大量企业私有知识 | RAG | 知识应放在检索系统里，不该硬塞进模型参数 |
| 需要实时信息或频繁更新 | RAG | 知识更新只需替换检索库 |
| 需要精确引用来源 | RAG | 检索可以明确知道信息来源 |
| 需要模型偏好（安全、有用性） | RL | RL 在偏好维度上推高上限 |
| 有确定性 rule reward 的任务（数学、逻辑） | RL（可跳过 SFT） | 规则明确时纯 RL 可能直接生效 |
| 简单格式调整、风格微调 | Prompt | 成本最低，迭代最快 |
| 功能验证和原型 | Prompt | 先看模型能不能做，再做其他决定 |

### RAG 优先的典型场景

以下场景应优先选择 RAG 而非 SFT：

- **企业知识问答**：产品文档、内部知识库、流程手册——这些信息量巨大且频繁变更，通过 SFT 无法有效注入。
- **长文档推理**：需要基于给定文档回答的问题。RAG 可以精确定位相关段落，SFT 只能依赖有限的训练数据。
- **多来源事实核查**：需要交叉验证多个来源的信息。RAG 可以检索并展示来源，SFT 只能输出训练时见过的"知识"。
- **用户具体数据**：每个用户有不同的上下文（邮件、聊天记录、代码库），无法预知的内容不适合 SFT。

### SFT 优先的典型场景

以下场景 SFT 的优势明显：

- **输出格式约束**：要求模型输出严格的 JSON/XML/代码格式，且格式复杂度高。
- **领域语言适应**：如法律文书、医学报告、技术文档等有特定表达规范的任务。
- **对话流程遵循**：客服场景中的多步处理流程、分类决策树等。
- **低延迟要求**：RAG 引入额外检索延迟，SFT 推理时无额外开销。

### SFT + RL 的递进关系

SFT 和 RL 不是二选一，而是递进关系：

- **SFT 给能力**：让模型学会遵循指令、输出特定格式和风格。
- **RL 推上限**：在 SFT 的基础上，通过偏好学习让模型在特定维度上做到更好。
- **先 SFT 再 RL**：如果 SFT 效果差，RL 基本不可能好——SFT 是模型能力的底座。
- **例外**：有确定性 rule reward 的任务（如数学竞赛题、代码正确性），可以跳过 SFT 直接做纯 RL。

## 常见误区

### "SFT 可以注入知识"

SFT 不适合用来注入知识。模型参数量有限，通过 SFT 塞进的知识量级远小于一个大型向量数据库。且知识更新需要重新训练，不灵活。知识应放在检索系统里，格式和风格通过 SFT 调整。

### "RAG 可以完全取代 SFT"

RAG 无法教会模型特定的输出格式或流程。如果任务的核心要求是输出特定结构（如严格的 JSON schema），RAG 只能提供内容，无法保证模型的输出格式。这种情况下 RAG + SFT 组合是最优解：SFT 保证格式，RAG 提供内容。

### "Prompt Engineering 是初级的，生产环境应该用 SFT"

很多生产级别的应用仅通过精心设计的 prompt + RAG 就达到了极好的效果。SFT 不是"prompt 的高级版"——它们解决不同的问题。优先用最简单的方案，只有在证明简单方案确实不够时才升级到更复杂的方案。

### "RL 比 SFT 更高级，所以应该跳过 SFT 直接做 RL"

这是最危险的一个误区。RL 建立在 SFT 的能力基础上——如果模型连正确的输出格式都不稳定，RL 阶段的探索-利用循环会让模型在错误的空间中徘徊。SFT 是地基，RL 是精装修。地基不稳的设计再精致也没有意义。

### "所有任务都满足 20% 规则"

20% 是一个经验阈值，不是科学精确值。对于某些创造性任务（如故事写作、头脑风暴），"正确率"的概念本身就不明确。但在有明确正确答案的任务上，20% 是一个经过验证的实用基准。如果 prompt 优化后模型表现依然极差，无论阈值如何，都不应期望 SFT 能创造奇迹。
