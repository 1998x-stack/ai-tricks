# 过拟合与欠拟合

## 是什么

**过拟合（Overfitting）**：模型在训练集上表现良好，但在验证集/测试集上表现差。训练 loss 低而验证 loss 高，表示模型"死记硬背"了训练数据，没有学到可泛化的规律。

**欠拟合（Underfitting）**：模型在训练集和验证集上都表现差。训练 loss 和验证 loss 都高，表示模型容量不足或训练不充分，连训练数据都没学好。

两者的本质区别在于泛化差距（Generalization Gap）的大小——即验证 loss 与训练 loss 的差值。

### 核心诊断指标

| 信号 | 训练 loss | 验证 loss | 诊断 |
|------|-----------|-----------|------|
| 过拟合 | 低 | 高 | 泛化差距大 |
| 欠拟合 | 高 | 高 | 两者都高 |
| 正常 | 低 | 接近训练 loss | 泛化良好 |
| 数据泄露 | 低 | 异常低 | 验证集可能被污染 |

## 为什么重要

**错误诊断导致错误措施，且代价高昂：**

- 把过拟合误诊为欠拟合，你会增大模型容量或加更多训练轮次——让问题更严重
- 把欠拟合误诊为过拟合，你会剪裁模型或加正则化——进一步降低模型容量，让欠拟合更严重
- 一次错误的方向调整意味着一整轮实验周期（几小时到几天）白费

泛化诊断是一切深度学习实验的第一步。每次实验前写假设，跑完后对比假设是否成立，这是系统提升实验效率的核心方法，参见 [[regularization]]。

## 如何使用

### 诊断步骤（每步确认后再下一步）

```
Step 1: 小数据过拟合测试（验证代码正确性）
  → 取 32~100 条训练样本，训练至训练 loss 降到底
  → 不能过拟合 → 检查：代码 bug、学习率、标签质量、数据预处理
  → 能过拟合 → 确认模型容量和代码无误

Step 2: 跑完整训练，监控训练/验证 loss 曲线
  → 看两条 loss 曲线的关系
  → 判断属于哪种模式

Step 3: 根据诊断结果执行对应行动方案
```

### 过拟合行动方案（泛化差距大）

优先级从高到低：

1. **增加数据量**：最有效的正则化手段。加数据（原始数据收集或数据增强）。数据增强优于任何参数调整，参见 [[../data-eval/data-eval]]
2. **增强数据增强强度**：Mixup、CutMix、RandAugment 等高级增强方法
3. **加大正则化参数**：Dropout rate、Weight Decay、Label Smoothing
4. **降低模型容量**：减少层数、隐藏单元数、降低 embedding 维度
5. **Early Stopping**：监控验证 loss，不下降即停止

**关键原则**：不要同时调整超过两个超参数。每次改一个变量，观察效果，得出结论，再改下一个。

### 欠拟合行动方案（两者都高）

优先级从高到低：

1. **验证代码正确性**：先确保能在小数据上过拟合
2. **增大模型容量**：加层、加隐藏单元、用更强的架构
3. **调整学习率**：先用 LR range test 找合适范围
4. **增加训练轮次**：可能只是训练不足
5. **检查数据质量**：标签噪声过大、数据预处理错误
6. **移除正则化**：暂时关掉 Dropout、减小 weight decay 排查是否是正则化过度

### 泛化差距量化参考

| 泛化差距（验证 loss - 训练 loss） | 含义 | 行动 |
|-----------------------------------|------|------|
| < 5% | 良好 | 维持 |
| 5%~15% | 轻微过拟合 | 轻量正则化 |
| 15%~30% | 明显过拟合 | 增加数据 + 更强正则化 |
| > 30% | 严重过拟合 | 系统性检查数据泄露/代码错误 |

> 注：不同任务和 loss 量级下这些阈值需要缩放调整，以上仅作参考框架。

## 常见误区

### 测试集 loss 降低不意味着没有过拟合

当负样本占比较大时，模型可能负样本学得特别好而正样本一塌糊涂——只看 loss 会被掩盖。必须同时检查 F1、AUC、混淆矩阵等分分类别指标。

### 验证集 loss 下降不代表泛化好

如果验证集切分不当，可能出现：
- **时间泄露**：验证集中包含训练集时间之后的数据（时序任务常见）
- **模板泄露**：验证集和训练集包含相似的重复模式
- **ID 泄露**：用户/物品 ID 在验证集和训练集中重复出现

验证集的可信度是所有诊断的前提。

### 特征要一个一个加

在特征工程中，每加一个特征都可能是过拟合的风险源。好的做法：每次加一个特征，跑完整实验，观察泛化差距变化。一次性加一堆特征导致某个特征导致过拟合也无从排查。

### 切勿同时调太多超参数

同时调整 Dropout 和 weight decay 和学习率，你无法知道哪个调整导致了效果变化。每次改动一个变量，保持其他不变。这是做可控实验的基本要求。

## 过拟合的一些反直觉场景

- 推荐系统中可能一个 epoch 就过拟合了——不像 CV/NLP 那样需要多轮训练才过拟合
- 大模型微调时，只要微调数据量小（几百条），几乎必定过拟合，不要惊慌
- 某些特征（如精确的用户 ID 特征）几乎必然导致过拟合
- 测试集上精度高但部署后效果差，可能是验证集本身就存在泄露
