# 梯度裁剪

## 是什么

梯度裁剪（Gradient Clipping）是一种在反向传播后、参数更新前插入的数值稳定技术。当梯度的范数（norm clipping）或每个分量（value clipping）超过预设阈值时，将其缩放到阈值以内，防止参数因为一次过大的梯度更新而跳出最优点或数值溢出。

PyTorch 提供两种方式：

```python
# 按范数裁剪（更常用，保留方向信息）
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)

# 按值裁剪（每个梯度分量独立截断）
torch.nn.utils.clip_grad_value_(model.parameters(), clip_value=0.5)
```

---

## 为什么重要

梯度爆炸是训练不收敛和 NaN 的常见原因，尤其在 RNN、Transformer、GAN 和深度 ResNet 中。梯度裁剪提供了一种"安全网"——即使梯度异常大，参数更新也不会失控。

然而，裁剪是一把双刃剑：

- **按范数裁剪**：会缩放所有参数的学习率，相当于在梯度异常时动态降低学习率。问题在于——如果梯度一直异常大，裁剪后所有参数都在用"打折"的梯度更新，可能导致收敛速度大幅下降。
- **按值裁剪**：更激进——每个梯度分量独立截断，这意味着不同参数的更新幅度被强制统一。对于某些需要大梯度的参数（如偏置项），值裁剪会破坏其学习能力。按值裁剪唯一适用的场景是极少数参数产生极端梯度而大部分正常。

在 RNN 类模型中，梯度爆炸是结构性问题（时间维度的连乘效应），**裁剪是必要条件**而非可选的锦上添花。对于 Transformer，裁剪更多是"保险丝"——正常情况下梯度不会爆炸，但极端 batch 或初始化不佳时需要兜底。

---

## 如何使用

### 阈值选择

- **1.0 是保守默认值**。来自 DeepSeek 和 LLaMA 等主流训练实践的推荐。大多数时候 1.0 能有效拦截异常梯度而对正常更新影响很小。对于不同规模的模型，观察分布后再调整比直接套用固定值更可靠。
- **0.2 是 PPO（强化学习）常用值**。PPO 的策略梯度本身就有极高的方差，训练过程中梯度范数的典型范围远大于监督学习。PPO 的裁剪更激进，因为策略更新过大就会破坏已有的采样。
- **0.5~5.0 是常见范围**。先用 1.0 试，观察裁剪触发比例——如果超过 30% 的 step 触发裁剪，说明阈值太低或梯度分布本身就有问题；如果从未触发，说明框架可能容忍不住异常。
- **一个有效策略**：先在部分数据上跑一次完整训练，记录每个 step 的梯度范数，再根据梯度范数的中位数和 90 分位数设定裁剪阈值。正常情况下，阈值应设置在梯度范数 90 分位数的 1.5-2 倍。

### 监控裁剪是否触发

仅仅加上裁剪代码是不够的——需要知道裁剪是否在频繁触发：

```python
total_norm = 0.0
for p in model.parameters():
    if p.grad is not None:
        total_norm += p.grad.norm().item() ** 2
total_norm = total_norm ** 0.5
if total_norm > max_norm:
    clip_count += 1  # 记录裁剪次数
```

如果裁剪触发率 > 30%，梯度分布有问题，单纯提高阈值或依赖裁剪是掩耳盗铃。

### 与混合精度的配合

在使用 `torch.cuda.amp` 时，梯度裁剪的时机至关重要：

```python
scaler.scale(loss).backward()
# 先 unscale，再裁剪
scaler.unscale_(optimizer)
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm)
scaler.step(optimizer)
scaler.update()
```

务必在 `unscale_` 之后、`step` 之前进行裁剪，否则裁剪会在缩放后的梯度上操作，达不到预期效果。

## 常见误区

- **"梯度裁剪能解决 NaN"**。裁剪只能处理**渐变型**NaN（梯度爆炸导致参数溢出），对**突发型**NaN（坏样本、log(0)）完全无效。详见 [[../debugging/nan-diagnosis]]。
- **"加了裁剪等于没加风险"**。裁剪改变了优化器的有效行为——对于 Adam 而言，裁剪不仅缩放当前 step 的梯度，还会影响动量估计（一阶矩和二阶矩）。频繁的裁剪会让 Adam 的动量估计值偏离真实分布，长期使用可能导致 optimizer 的状态退化。
- **"值裁剪比范数裁剪安全"**。值裁剪每个分量独立截断，破坏了梯度的方向信息，相当于在梯度的每个分量上做不同的缩放。范数裁剪保留了方向，仅在幅度上做全局缩放，对优化路径的扰动更小。值裁剪唯一合理的场景是已知只有极个别参数会产生异常梯度（如视频模型的某些 "死亡" 神经元）。
- **"阈值越大越安全"**。阈值过大会让裁剪形同虚设——梯度范数 100 而阈值 1000，从未触发则毫无作用。阈值太小则会频繁触发，拖慢收敛。正确的做法是根据训练初期的梯度分布动态设定。
- **"裁剪是调参的基本工具，应该始终开启"**。对于排查阶段，裁剪会掩盖问题（如坏样本产生的 NaN 梯度）。小数据过拟合测试时应当关闭裁剪，暴露问题后再决定是否启用。详见 [[../debugging/small-data-overfit-test]]。
- **"裁剪只影响训练过程不影响最终模型"**。频繁裁剪会系统性地改变参数分布，尤其是偏置项——裁剪倾向于减小参数更新幅度，在深度网络中偏置的梯度往往远大于权重，裁剪总是先打到偏置的梯度，导致偏置更新不足。
