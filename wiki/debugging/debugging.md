# 调试与故障排查

## 概述

深度学习调试的核心是系统化排除法：先验证模型能否在小数据上过拟合和随机标签上不过拟合，再按优先级逐一排查学习率、数据流水线、损失函数和代码实现。不要怀疑神经网络的能力 —— 当模型连小数据都无法拟合时，问题一定出在代码、数据或超参数上。

## 详细知识

- [[nan-diagnosis]]: 训练中loss突变为NaN或逐渐发散到NaN的系统化排查方法，包括检测输入数据、loss计算和梯度爆炸。
- [[small-data-overfit-test]]: 拿32-100条样本让模型过拟合，快速验证代码、数据和训练流程是否正确的最有效手段。
- [[gradient-clipping]]: 通过裁剪梯度范数防止梯度爆炸导致NaN，是训练不稳定的临时修复手段，但应优先定位根因。
- [[training-curve-diagnosis]]: 通过观察训练集和验证集的loss曲线关系判断过拟合、欠拟合或学习率问题，指导调参方向。

## 核心技巧

### 小数据过拟合测试
- **来源**: Andrej Karpathy / BugBuster喵 / TNotes / Captain Jack
- **适用场景**: 任何深度学习模型首次训练
- **原文**: "先上小规模数据，模型往大了放，只要不爆显存，能用256个filter你就别用128个。直接奔着过拟合去。"
- **要点**: 拿32-100条样本，让模型把训练loss打到非常低。如果做不到，别谈调参 —— 大概率是标签错了、输入没对齐、mask有问题、loss写错了、学习率离谱、数据增强把信号破坏了、训练和推理的tokenizer不一致。
- **参见**: [[../data-eval/data-eval]], [[../optimizer-lr/optimizer-lr]]

### 随机标签测试
- **来源**: BugBuster喵
- **适用场景**: 验证模型实现是否正确
- **原文**: "随机标签能不能不过拟合"
- **要点**: 打乱标签后模型应该无法过拟合（loss不下降）。如果随机标签下loss依然下降，说明模型在记忆数据而非学习规律，通常是数据泄露或代码bug。
- **参见**: [[../data-eval/data-eval]]

### 五种最常见的实现Bug
- **来源**: Josh Robin (伯克利) / 量子位
- **适用场景**: 调试新搭建的模型
- **原文**: "错误的张量形状；预处理输入错误；损失函数错误输入；忘记设置正确的训练模型；错误的数据类型。"
- **要点**: 1) 张量形状不匹配；2) 预处理/输入错误；3) 损失函数输入错误（如输入了未归一化的logits而非softmax输出）；4) 忘记设置 model.train()；5) 数据类型错误（未转成 float32）。
- **参见**: [[../normalization/normalization]]

### NaN诊断：突然出现的NaN
- **来源**: TNotes / 陈城南
- **适用场景**: 训练到中途loss突然变成NaN
- **原文**: "输入中存在Nan（坏样本）；损失函数的计算,如交叉熵损失函数的计算可能出现log(0),所以就会出现loss为Nan"
- **要点**: 首先检查输入数据是否含有NaN，其次检查loss计算中是否有log(0)操作。使用 `torch.isnan()` 逐层排查输入和中间结果。
- **参见**: [[../optimizer-lr/optimizer-lr]]

### NaN诊断：逐渐发散到NaN
- **来源**: TNotes / 陈城南 / 爱睡觉的KKY
- **适用场景**: Loss逐渐增大然后变成NaN
- **原文**: "学习率过高；梯度爆炸"
- **要点**: 学习率太大导致参数更新越过最优点；梯度爆炸导致参数数值溢出。先降低学习率，若无效则启用梯度裁剪作为临时修复。
- **参见**: [[../optimizer-lr/optimizer-lr]], [[optimizer-lr#梯度裁剪]]

### torch.autograd.detect_anomaly()
- **来源**: TNotes
- **适用场景**: 定位产生NaN的具体操作
- **原文**: "代码问题定位 autograd.detect_anomaly()"
- **要点**: 用 `with torch.autograd.detect_anomaly():` 包裹前向和反向传播，PyTorch会在产生NaN的精确位置抛出异常，精确定位问题操作。
- **参见**: [[debugging#NaN诊断：突然出现的NaN]]

### 梯度裁剪（临时修复）
- **来源**: 陈城南 / 爱睡觉的KKY
- **适用场景**: Loss NaN或梯度爆炸时的临时措施
- **原文**: "如果实在无法解决，梯度CLIP防止NaN也行，但这样可能治标不治本"
- **要点**: 梯度裁剪可以防止NaN继续训练，但可能掩盖真正的问题。有时裁剪会导致性能下降 —— 正常收敛的模型加clip反而变差。建议定位根因而非依赖clip。
- **参见**: [[../optimizer-lr/optimizer-lr]]

### 过拟合诊断
- **来源**: 无忧无虑 / 四点半的理一
- **适用场景**: 训练集loss低，验证集loss高
- **原文**: "在训练集上的误差很小...但是验证集的loss相对较大...那就是过拟合了"
- **要点**: 训练loss低（如<0.01）但验证loss高（如0.3-0.6）。解决方案优先级：加数据 > 加正则化（dropout, weight decay）> 降低模型容量。最有效的方法永远是多标记数据。
- **参见**: [[../regularization/regularization]], [[../data-eval/data-eval]]

### 欠拟合诊断
- **来源**: 无忧无虑 / 四点半的理一
- **适用场景**: 训练集和验证集loss都高
- **原文**: "在训练集和验证集上的loss都比较大，都大于0.3，那可能就是欠拟合了"
- **要点**: 两边loss都高，说明模型表达能力不足或学习率不对。先检查学习率，然后增大模型宽度/深度，或检查代码bug。
- **参见**: [[../optimizer-lr/optimizer-lr]]

### 训练不收敛：检查数据流水线
- **来源**: 陈城南 / BugBuster喵
- **适用场景**: Loss不下降或震荡
- **原文**: "dataset构建基本不会有啥问题，构建好了后需要检查一下输出的数据读取是否正确"
- **要点**: 把dataloader单独跑一遍，用 `torch.isnan()` 检测源数据是否正确。检查数据增强是否破坏了信号（如CV任务裁剪掉了目标区域），检查label与input是否对齐。
- **参见**: [[../data-eval/data-eval]]

### 训练不收敛：检查优化器参数
- **来源**: 陈城南 / 摘星狐狸
- **适用场景**: 模型参数未参与训练
- **原文**: "要保证model的参数被加载到optimizer里了...如果你的model2的参数未添加到optimizer里，这部分参数就没参与学习"
- **要点**: 确保所有模型参数都被传入了优化器。先后创建model1、optimizer、model2时，model2的参数不会自动被optimizer管理。
- **参见**: [[../optimizer-lr/optimizer-lr]]

### 损失函数排查
- **来源**: 陈城南 / Captain Jack
- **适用场景**: Loss异常但数据/model无问题
- **原文**: "loss是基本不会出错的地方，但loss也是最容易出错的地方"
- **要点**: 排查loss函数本身是否有 log(0) 等操作。多任务loss要确保各loss在同一量级。分类用Softmax，回归用L2但要注意输出值的范围 —— 预测10000而模型输出0会导致NaN。
- **参见**: [[../optimizer-lr/optimizer-lr]]

### 先调学习率
- **来源**: 无忧无虑 / 四点半的理一 / 爱睡觉的KKY
- **适用场景**: 模型效果不理想时的第一步
- **原文**: "优先调learning rate! 优先调learning rate! 优先调learning rate!"
- **要点**: 学习率是影响最大的超参数。先用LR Range Test找到合理范围：从1e-5开始以10倍递增，观察loss曲线，选择发散前的那一档。不要同时调超过两个超参数。
- **参见**: [[../optimizer-lr/optimizer-lr]]

### 学习率预热扫描（LR Range Test）
- **来源**: 四点半的理一
- **适用场景**: 寻找合适的初始学习率
- **原文**: "从1e-5开始，以10倍为单位往上跳，比如1e-4、1e-3、1e-2，观察loss曲线"
- **要点**: Loss应平稳下降；从某个点开始loss发散/爆炸时，前一个值就是候选学习率。也可以用指数递增的方式在一个epoch内扫完。
- **参见**: [[../optimizer-lr/optimizer-lr]]

### 可视化调试：Layer Activations
- **来源**: 杨军 (CS231n)
- **适用场景**: 理解神经网络内部状态
- **原文**: "比较理想的layer activation应该具备sparse和localized的特点"
- **要点**: 可视化隐藏层的activation矩阵。如果feature map的activation跟原始输入长得一样，说明这个filter没学到有用的东西。理想的activation应该是稀疏且局部化的。
- **参见**: [[../cnn-cv/cnn-cv]]

### 可视化调试：Layer Weights
- **来源**: 杨军 / Captain Jack
- **适用场景**: 判断训练是否正常
- **原文**: "期望的良好的卷积层的weight可视化出来会具备smooth的特性"
- **要点**: 第一层卷积的weight可视化应该是平滑的。出现大量噪点说明训练过程有问题。但小filter（如3x3）只有9个像素，smooth与否不一定有意义。
- **参见**: [[../cnn-cv/cnn-cv]]

### 可视化调试：Softmax输出分布
- **来源**: Captain Jack
- **适用场景**: 判断分类网络是否学习充分
- **原文**: "刚开始的网络预测都是在0.5上下，很模糊。随着学习过程，网络预测会慢慢的移动到0,1这种极值附近"
- **要点**: 分类网络从含糊到清晰。如果预测概率分布集中在中间值（如0.5附近），说明网络还没学充分，需要继续训练或调整超参数。
- **参见**: [[../optimizer-lr/optimizer-lr]]

### 对比训练集和验证集的Loss
- **来源**: Captain Jack / 无忧无虑
- **适用场景**: 判断过拟合、训练是否充分、是否需要early stop
- **原文**: "判断过拟合，训练是否足够，是否需要early stop的依据"
- **要点**: 监控train/val loss的差距。差距大 = 过拟合；都高 = 欠拟合。准确率可能突变（长期为0突然变1），loss更稳定可靠。
- **参见**: [[../regularization/regularization]]

### 不要怀疑NN的能力
- **来源**: Captain Jack
- **适用场景**: 模型在小数据上也无法拟合时的心理建设
- **原文**: "不要怀疑NN的能力，不要怀疑NN的能力，不要怀疑NN的能力。就我们调参狗能遇到的问题，NN没法拟合的，这概率是有多小？"
- **要点**: 如果模型连小数据都无法过拟合，99%是代码bug或数据问题，而不是神经网络的表达极限。检查dataloader、label、loss实现。
- **参见**: [[debugging#小数据过拟合测试]]

### 不要盲目信任论文默认参数
- **来源**: Captain Jack / BugBuster喵
- **适用场景**: 使用开源代码和预训练模型
- **原文**: "不要完全相信论文里面的东西。结构什么的觉得可能有效果，可以拿去试试。"
- **要点**: 开源repo的默认配置往往是为跑通demo设计的，不一定适合你的数据和任务。batch、lr、max length等都需要重新调试。
- **参见**: [[../optimizer-lr/optimizer-lr]]

### 一次只改一个变量
- **来源**: 四点半的理一 / BugBuster喵
- **适用场景**: 调参实验设计
- **原文**: "不要同时调超过两个超参数。这是新手和大神最本质的区别。"
- **要点**: 同时改多个参数无法归因。每次实验只改一个变量，记录所有配置。效果涨了知道为什么涨，跌了知道谁背锅。
- **参见**: [[../data-eval/data-eval]]

### 记录一切
- **来源**: BugBuster喵 / 四点半的理一
- **适用场景**: 实验管理和复现
- **原文**: "每次实验至少记录代码版本、数据版本、配置、随机种子、硬件、训练时间、最优checkpoint、评估结果、错误样例"
- **要点**: 记录seed、config、数据版本、代码版本。固定随机种子（42/3407）。别相信自己的记忆，两周后你只会记得"大概好像那组不错"。
- **参见**: [[../data-eval/data-eval]]

### 先诊断再下手（消融思维）
- **来源**: 四点半的理一 / BugBuster喵
- **适用场景**: 调参前的系统诊断
- **原文**: "真正高效的调参，是先诊断，再下手。你的模型到底卡在哪里？"
- **要点**: 训练集loss高 → 模型表达能力或学习率问题；训练集loss低但验证集高 → 过拟合，加数据/正则化；两个都低但验证集不收敛 → 学习率进入悬崖区。
- **参见**: [[../optimizer-lr/optimizer-lr]], [[../regularization/regularization]]

### 连续调参无提升时去看数据和错误样例
- **来源**: BugBuster喵
- **适用场景**: 多组实验都无法提升
- **原文**: "当你连续调10组参数都没有稳定提升时，别再调第11组，去看数据和错误样例。"
- **要点**: 调参瓶颈往往不在参数本身，而在数据质量、标签噪声、评估方式。做错误分析比盲调参数更有价值。
- **参见**: [[../data-eval/data-eval]]

### 确认模型加载了预训练权重
- **来源**: 陈城南
- **适用场景**: 使用预训练模型微调
- **原文**: "使用pretrained模型时记得把load过程打印出来，保证模型被load了"
- **要点**: 加载预训练权重后务必打印加载状态，确认所有参数都被正确加载。权重未加载但训练继续的情况比想象中更常见。
- **参见**: [[../initialization/initialization]]

### Debug流程：从简单到复杂
- **来源**: Josh Robin (伯克利) / 量子位
- **适用场景**: 系统的深度学习调试流程
- **原文**: "从不最简单模型入手；成功搭建模型，重现结果；分解偏差各项，逐步拟合数据"
- **要点**: 1) 从最简单的架构开始；2) 在小数据上跑通；3) 逐步增加复杂度；4) 由粗到细搜索超参数；5) 欠拟合增大模型，过拟合加数据。
- **参见**: [[../optimizer-lr/optimizer-lr]], [[../data-eval/data-eval]]

### 调试时要悲观主义
- **来源**: Josh Robin (伯克利) / 量子位
- **适用场景**: 调试心态
- **原文**: "调试深度学习模型的第一要义——悲观主义"
- **要点**: 先假设你的代码有bug，逐步验证每个环节。不要假设代码正确然后怀疑模型能力。
- **参见**: [[debugging#不要怀疑NN的能力]]

### 观察Loss胜于观察准确率
- **来源**: Captain Jack
- **适用场景**: 训练过程中的监控
- **原文**: "准确率虽然是评测指标，但是训练过程中还是要注意loss的。你会发现有些情况下，准确率是突变的"
- **要点**: Loss提供连续稳定的信号，准确率可能突然跳变。不要因为前期准确率低就提前终止训练。
- **参见**: [[../optimizer-lr/optimizer-lr]]

### 确认训练/推理预处理一致性
- **来源**: BugBuster喵
- **适用场景**: 训练效果好但推理效果差
- **原文**: "训练和推理的预处理是否完全一致"
- **要点**: 训练时用的预处理（归一化参数、resize方式、数据增强）必须和推理时完全一致。不一致会导致严重的性能下降，且难以排查。
- **参见**: [[../data-eval/data-eval]]

### 梯度检查与裁剪
- **来源**: 摘星狐狸
- **适用场景**: 防止梯度爆炸/消失
- **原文**: "梯度归一、梯度裁剪一定很重要"
- **要点**: 计算梯度范数，设置阈值进行裁剪。PyTorch中使用 `torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm)`。经常与bf16混合精度配合使用。
- **参见**: [[../optimizer-lr/optimizer-lr]]

### 早停策略
- **来源**: BugBuster喵 / 爱睡觉的KKY
- **适用场景**: 防止过拟合和节省算力
- **原文**: "不要过早的early stopping，有时候收敛平台在后段"
- **要点**: 先确保模型能在训练集上过拟合，再用早停防过拟合。有些模型前期看不出起色，然后开始稳定学习。给NN一点时间。
- **参见**: [[../regularization/regularization]]

### 可视化Embedding（t-SNE）
- **来源**: 杨军
- **适用场景**: 观察模型对数据的整体理解
- **原文**: "通过t-SNE对隐藏层进行降维...人工review同一类图片在降维之后的低维空间里是否处于相邻的区域"
- **要点**: 用t-SNE将隐藏层输出降到2D/3D，查看同类样本是否聚类。可用于发现数据问题或模型学习问题。先用PCA降维再用t-SNE效果更好。
- **参见**: [[../cnn-cv/cnn-cv]]

### 数据泄露排查
- **来源**: BugBuster喵
- **适用场景**: 离线指标好但线上效果差
- **原文**: "验证集是随机切的，同一个用户的连续行为同时出现在训练和验证里，模型学到了一些用户短期偏好泄漏"
- **要点**: 按时间切分而非随机切分可避免数据泄露。同样模板/用户/场景的数据不要同时出现在训练和验证集中。数据泄露是假提升的最大来源。
- **参见**: [[../data-eval/data-eval]]

### 验证集被反复调穿
- **来源**: BugBuster喵
- **适用场景**: 长期调参项目管理
- **原文**: "每天看验证集调参数，调到最后验证集也成了训练的一部分"
- **要点**: 保留一个真正不碰的最终测试集，重要版本才评一次。验证集被"调穿"后不再代表泛化能力。
- **参见**: [[../data-eval/data-eval]]

### 梯度累积与显存优化
- **来源**: LindgeWAI / 爱睡觉的KKY
- **适用场景**: 显存不足时模拟大批量训练
- **原文**: "梯度累积：增大batch，规避GPU内存的限制"
- **要点**: 在小batch上累积梯度后再更新参数，等效于大batch。注意loss要除以accumulation_steps。对表示学习/对比学习领域，batch size一般越大越好。
- **参见**: [[../optimizer-lr/optimizer-lr]]
