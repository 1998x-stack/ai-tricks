# 小数据过拟合测试

## 是什么

小数据过拟合测试是深度学习调试中**优先级最高的系统验证**：拿 32-100 条训练样本，让模型撑到最大容量（不加任何正则化），目标是在训练集上把 loss 打到接近零。如果模型连这么少的数据都无法过拟合，说明代码、数据或超参数有严重 bug，不应继续上全量数据或调参。

Karpathy 有一句广为流传的建议："先上小规模数据，模型往大了放，只要不爆显存，能用 256 个 filter 你就别用 128 个。直接奔着过拟合去。"

---

## 为什么重要

这条测试之所以被反复强调，是因为它用最低的成本覆盖了最多的故障点：

- **验证代码实现正确**。前向传播、loss 计算、反向传播、参数更新这四个环节中任何一个出错，小数据过拟合都会失败。
- **验证数据流水线正确**。label 和 input 是否对齐、数据增强是否破坏了信号、预处理是否与推理一致。
- **验证优化器配置正确**。参数是否全部传入 optimizer、学习率是否在合理范围。
- **建立性能基线**。小数据上的收敛速度和损失值成为后续调试的参考点——如果全量数据时 loss 形状与基线不同，说明 scaled 后的系统有问题。

> **核心原则**：不要怀疑神经网络的能力。如果模型连小数据都无法过拟合，99% 是代码 bug 或数据问题。

---

## 如何使用

### 标准流程

1. **准备数据子集**。从训练集中随机抽取 32-100 条样本。保持类别分布大致均衡即可，不需要严格分层采样。
2. **去除所有正则化**。关闭 Dropout、Weight Decay（设为 0）、Label Smoothing、数据增强。目的是让模型全力记忆。
3. **增大模型容量**。确保模型有足够容量——层数或宽度设为正常值的 1.5-2 倍，或者直接用最宽的版本。
4. **使用合理的学习率**。建议从 1e-3 或 3e-4 开始。如果 loss 发散，降到 1e-4；如果 loss 下降太慢，提到 1e-2。
5. **训练足够长的时间**。NN 在小数据上可能需要数百甚至上千个 epoch 才能完全过拟合，不要跑了 10 步就下结论。
6. **观察 loss 是否趋近于零**。对于分类任务，loss 应接近 0（或等价于 100% 准确率）；对于回归任务，loss 应非常接近 0。
7. **检查输出**。不仅看 loss，还检查模型输出是否含 NaN（用 `torch.isnan()` 验证）。

### 通过/失败的判定

| 现象 | 结论 | 下一步 |
|------|------|--------|
| loss 稳定下降到 0 附近 | 通过 | 可以上全量数据 |
| loss 下降但不为 0 | 可能容量不够或学习率偏低 | 增大模型或调高 lr |
| loss 震荡不下降 | 学习率太高或代码 bug | 降低 lr 到 1e-5 试 |
| loss 变成 NaN | 突发型或渐变型 NaN | 见 [[../debugging/nan-diagnosis]] |
| loss 先降后升到 NaN | 学习率太高导致梯度爆炸 | 降低 lr，启用 [[../debugging/gradient-clipping]] |
| loss 完全不变 | 参数未更新 | 检查 optimizer 是否包含所有参数 |

## 随机标签变体

小数据过拟合测试的"对照组"——打乱标签后，模型应该**不能**过拟合。

### 操作步骤

1. 复制之前用的小数据子集（32-100 条）。
2. 彻底打乱 label：`labels = labels[torch.randperm(len(labels))]`。
3. 用完全相同的配置（大模型、无正则化）训练。
4. 观察 loss。

### 结果解读

- **随机标签下 loss 不下降**：正常。模型无法在随机映射中找到规律，保持在接近随机猜的 loss 水平。
- **随机标签下 loss 依然下降**：**严重警告**。说明模型在"记忆"数据而非学习规律。常见原因：数据泄露（label 信息存在于 input 中）、代码 bug（label 与 input 对齐错误导致模型学到了偶然相关性）、过强的数据增强把信号破坏后又让模型记住了噪声。

### 何时跳过

以下情况下随机标签测试的结论需要谨慎解读：

- 模型容量不足以记忆随机标签（小模型 + 大数据量不会过拟合随机标签，这不代表你的代码就正确）。
- 任务本身有确定性规则（如排序任务），打乱后的熵不足以完全破坏结构。
- 回归任务的 label 空间是连续的，打乱后模型依然可能学习到输入到输出的某种映射。

## 常见误区

- **"32 条样本过拟合太简单，测不出问题"**。恰恰相反——正是在小数据上，数据预处理错误、label 偏移、mask 疏漏这些隐蔽问题才会立刻暴露。全量数据时信号冗余可能把这些 bug 稀释掉，但效果始终上不去。
- **"跑了 10 个 epoch loss 没降，模型有问题"**。NN 在小数据上也可能需要上百个 epoch 才能从随机初始化进入学习状态。尤其是 Transformer，前期收敛非常慢。
- **"小数据过了就行，全量数据应该也会有类似表现"**。小数据通过说明代码和数据流水线基本正确，但全量数据可能引入新的问题（数据分布变化、采样偏差、batch 统计量差异）。小数据测试是**必要条件，不是充分条件**。
- **"测试时数据增强不能去掉，否则不是真实场景"**。过拟合测试的目的是验证模型训练，而非验证数据增强。先关掉增强确认模型能力，再逐步加回增强。
- **"一次使用了 1000 条样本，也算小数据"**。"小"意味着能让模型在合理时间内完全记忆。1000 条对于大模型（上亿参数）确实算小，但对于小模型（几万参数）可能已经需要很长时间。根据模型容量动态调整子集大小。
