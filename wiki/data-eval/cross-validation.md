# 交叉验证设计

## 是什么

交叉验证（Cross-Validation）是一种评估模型泛化能力的重采样技术，核心思想是将数据集划分为互补的子集，在部分子集上训练、在剩余子集上验证，并轮换角色以获取稳定的性能估计。

常见的交叉验证形式包括：

- **K-Fold CV**：将数据均分为K份，每次用K-1份训练、1份验证，重复K次。K通常取5或10。
- **Stratified K-Fold**：保持每折中类别比例与全集一致，适用于分类任务。
- **Group K-Fold**：确保同一组的样本不会同时出现在训练和验证集中，适用于含用户ID或会话ID的数据。
- **Leave-One-Group-Out（LOGO）**：每次留出一个组做验证，适用于组数量不多但每组样本量大的场景。
- **Time Series CV**（时序交叉验证）：按时间顺序递增地扩展训练窗口，验证窗口始终在训练窗口之后。常见的实现包括滑动窗口（固定窗口大小）和扩展窗口（累积历史数据）。
- **Leave-One-Out（LOO）**：每次留一个样本做验证，适用于极小数据集。方差大且计算成本高，不推荐用于超过100条样本的数据集。
- **Repeated K-Fold**：重复多次K-Fold，每次使用不同的随机种子，进一步降低估计方差。当数据量小于1000条时推荐使用。
- **Nested CV**（嵌套交叉验证）：外层循环评估模型性能，内层循环做超参数选择。内外层分别使用不同的K值，避免超参数选择过程中的信息泄漏。适用于研究场景中的无偏性能估计。

## 为什么重要

交叉验证的价值主要体现在三个方面：

1. **更可靠的性能估计**：单次固定验证集的评估可能因偶然切分而产生较大偏差。例如，随机切分可能恰好把最容易的样本分到验证集，造成性能高估。交叉验证通过多次评估取平均，大幅降低了评估方差，使我们对模型的真实性能有更准确的把握。

2. **防止验证集过拟合**：在固定验证集上反复调参，验证集的信息会无形中渗入模型选择过程——你记住了一个特定的验证集上哪些参数效果好，下次调参时无意中偏向这些参数。最终导致验证集无法代表真实泛化。交叉验证（尤其是结合留一最终测试集）可有效缓解这一问题。

3. **检测数据稳定性与诊断问题**：如果不同折之间的性能波动很大（如某折F1=0.95、另一折F1=0.60），这本身就是重要的诊断信号。可能的原因包括：数据分布不稳定（不同时间段数据特征不同）、存在未被注意到的分组结构、标签噪声集中在特定时间段、或者数据量过小导致随机性主导评估结果。跨折稳定性和[[../debugging/debugging|调试]]的起点。

## 如何使用

### 黄金原则：切分方式必须与任务结构对齐

**不要默认使用随机切分。** 切分方式的选择取决于数据的生成机制和任务结构，这是交叉验证设计中最重要的决策。

**时序数据必须按时间切分。** 用前N天的数据训练，预测第N+1天；再用前N+1天训练，预测第N+2天，依此类推。随机切分时序数据会让模型"看到未来"——训练集中包含了未来信息用于预测"过去"，导致离线指标严重高估线上效果。一个真实案例：某电商搜索排序模型离线AUC提升了0.7个点，团队很兴奋；上线后没有任何动静，甚至部分品类指标还下降了。排查发现验证集是随机切的，同一用户的连续行为同时出现在训练和验证中，模型学到了用户短期偏好泄漏。

**含分组结构的数据必须按组切分（Group K-Fold）。** 例如推荐系统的用户ID、医疗数据中的患者ID、对话数据中的会话ID、多张图片来自同一个人的姿态估计数据。同一用户/患者的多个样本若同时出现在训练和验证集中，模型可能学到用户级短期偏好或个体特征（如"这个用户喜欢点击红色按钮"），造成信息泄漏，使离线指标无法反映真实泛化能力。

**模板化数据必须按模板切分。** 当数据由模板自动生成（如客服话术、格式化报表），相同模板的不同实例会高度相似。若不按模板ID切分，随机拆分会使验证集包含与训练集几乎相同的模板实例，导致泛化能力被高估。这种情况下应[[../data-eval/data-eval|按话术模板和时间切分]]。

### 具体操作

```python
# 时序数据：不可使用 KFold，必须使用 TimeSeriesSplit
from sklearn.model_selection import TimeSeriesSplit
tscv = TimeSeriesSplit(n_splits=5)
for train_idx, val_idx in tscv.split(X):
    # train_idx 始终在时间上早于 val_idx
    ...

# 含分组数据：使用 GroupKFold
from sklearn.model_selection import GroupKFold
gkf = GroupKFold(n_splits=5)
# groups 是每个样本所属的分组 ID（如用户ID、会话ID）
for train_idx, val_idx in gkf.split(X, y, groups=user_ids):
    ...

# 分层+分组：使用 StratifiedGroupKFold（sklearn >= 1.0）
from sklearn.model_selection import StratifiedGroupKFold
sgkf = StratifiedGroupKFold(n_splits=5)
for train_idx, val_idx in sgkf.split(X, y, groups=user_ids):
    ...
```

### K值选择指南

- **K=5**：实践经验中最常用的选择，兼顾偏差与方差。验证集占20%，对大多数数据集而言足够评估。是默认首选。
- **K=10**：验证集占10%，评估结果偏差更小但计算成本高一倍。适用于数据量中等（1000-10000条）且评估精度要求高的场景。
- **K=3**：验证集占33%，训练数据减少较多，适用于数据量很大（10万+）且计算资源有限的场景。
- **K=N（LOO）**：仅适用于极少量数据（<100条），偏差最小但方差极大，且计算成本为O(N)。
- **小数据集（<1000条）**：推荐 Repeated K-Fold（如重复5次5折），用多次随机化降低评估方差。

### 评估指标跨折聚合

交叉验证的每折都会产生一个评估指标值。如何聚合这些值影响最终结论：
- **均值 ± 标准差**：最常用的报告方式。如"F1 = 0.87 ± 0.03"。标准差大说明模型对数据切分敏感，需要排查上述诊断信号。
- **不推荐**：将所有折的预测结果拼接后统一计算指标（俗称的"micro-average across folds"）。这会引入不同折的分布差异，且无法反映跨折的稳定性信息。

## 常见误区

### 误区一：时序数据用随机切分

这是最常见的致命错误。时序数据存在时间依赖性，用随机切分会使未来的数据出现在训练集中，模型在"预测过去"——离线指标虚高，上线后效果断崖式下跌。解决方案：[[#具体操作|使用TimeSeriesSplit]]。

### 误区二：忽略分组结构导致信息泄漏

推荐场景中，同一个用户的多个行为样本可能被分配到不同折，模型会学到用户级特征（如点击习惯）而非真正的泛化模式。解决方案：[[#具体操作|使用GroupKFold按用户切分]]。

### 误区三：K值设得太大或太小

K太小（如K=2）导致每次只有一半数据训练，模型欠拟合，评估偏差大。K太大（如K=20或LOO）导致各折训练集之间高度重叠（每次只少1/K的数据），评估方差小但计算成本爆炸，且偏差可能增大。K=5是大多数场景的安全起点。

### 误区四：交叉验证替代最终测试集

即使做了交叉验证，仍然需要保留一个完全不碰的最终测试集。交叉验证的结果应该作为模型选择的依据，最终测试集仅在完成所有调参后做一次评估。[[../data-eval/data-eval|验证集上反复调参会使其成为训练的一部分]]——交叉验证不能免除这一规律。

### 误区五：不平衡数据中用默认K-Fold

类别极度不平衡时，随机K-Fold可能导致某些折完全没有正样本。应始终使用Stratified K-Fold或在分组场景中使用最近的StratifiedGroupKFold来保持每折和训练/验证间的类别分布一致。

### 误区六：交叉验证结果负相关不调查

如果发现某些折上模型表现极差而另一些折很好，不要仅理解为"方差大"。应调查差的那几折有什么共性——是不是某个时间段的数据特别难？是不是某种类型样本在这些折中更集中？这个调查本身就是对数据理解的深化。
