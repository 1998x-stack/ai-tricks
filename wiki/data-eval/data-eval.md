# 数据与评估

## 概述

数据质量与评估方法决定了模型调参的方向是否正确——在脏数据或错分的验证集上调参，所有"提升"都是假提升。本章涵盖数据清洗、数据分布匹配、交叉验证设计、评估指标选择、训练/验证/测试集切分、数据增强、类别不平衡处理、标签噪声检测、数据版本管理、以及LLM评测等核心话题。

## 详细知识

- [[cross-validation]]: 交叉验证是评估模型泛化能力的基础方法，需根据任务类型（时序、用户ID等）选择适当的切分策略，避免信息泄漏。
- [[focal-loss]]: 通过调制系数降低易分类样本的loss权重，使训练聚焦于难样本，对类别极度不平衡的分类任务有奇效。
- [[data-augmentation]]: 通过裁剪、翻转、旋转、色彩抖动等CV操作或同义词替换、回译等NLP操作扩充训练数据，提升模型泛化能力。
- [[class-imbalance]]: 处理分类任务中各类别样本数量差异巨大的问题，常用方法包括Focal Loss、类别加权、过采样和欠采样。
- [[eval-metrics]]: 评估指标选择决定调参方向是否正确，分类看F1/AUC，排序看NDCG，检测看mAP，需与业务目标对齐。

## 核心技巧

### 先在小数据上过拟合，确认代码无误，再上全量
- **来源**: 爱睡觉的KKY / Captain Jack / BugBuster喵
- **适用场景**: 任何深度学习项目的初始阶段
- **原文**: "先overfit 再trade off，首先保证你的模型capacity能够过拟合，再尝试减小模型，各种正则化方法" / "刚开始，先上小规模数据，模型往大了放，只要不爆显存，能用256个filter你就别用128个。直接奔着过拟合去。"
- **要点**: 拿100条甚至32条样本，让模型把训练loss打到非常低。如果这都做不到，别谈调参——大概率是标签错了、输入没对齐、loss写错了、学习率离谱。
- **参见**: [[../debugging/debugging]]

### 数据质量优先于模型调参
- **来源**: 白小鱼 / BugBuster喵
- **适用场景**: 任何机器学习项目，尤其是数据收集阶段
- **原文**: "机器学习的效果很大程度上取决于参数调节的好坏……不过，上述说法很明显忽略了一个重要的条件，那就是，所用来训练的数据是行之有效的" / "当你连续调10组参数都没有稳定提升时，别再调第11组，去看数据和错误样例。"
- **要点**: 如果数据不足以代表真实分布、标签不准确、或存在样本偏差，再好的参数也无效。先花时间检查数据，再做调参决策。
- **参见**: [[label-noise]], [[../debugging/debugging]]

### 训练集分布必须覆盖测试集分布
- **来源**: Tang AI
- **适用场景**: 分类/检测/生成任务，尤其是数据收集阶段
- **原文**: "数据的多样性非常重要，再三强调，训练集的数据分布一定要包括测试集的数据分布，这样测试集才会有好的效果。"
- **要点**: 测试集效果差但训练集还可以时，检查测试集里有没有训练集未覆盖的数据类型。人工针对性添加缺失类型的样本（5-10条起步）。
- **参见**: [[data-augmentation]]

### 交叉验证方式必须结合任务设计，时序数据防未来泄漏
- **来源**: 爱睡觉的KKY / BugBuster喵
- **适用场景**: 时序预测、推荐系统、风控模型
- **原文**: "cross validation方式要结合任务设计，数据标签设计，其中时序数据要避免未来信息泄漏" / "验证集是随机切的，同一个用户的连续行为同时出现在训练和验证里，模型学到了一些用户短期偏好泄漏。重新按时间切分以后，离线提升直接没了一半。"
- **要点**: 时序数据必须按时间切分，不能用随机切分；推荐系统等含用户ID的数据要按用户切分，避免同一用户行为横跨训练/验证集导致信息泄漏。
- **参见**: [[train-val-test-split]], [[time-series]]

### 训练/验证/测试集切分方式决定实验可信度
- **来源**: 无忧无虑 / BugBuster喵
- **适用场景**: 所有监督学习任务
- **原文**: "调参请在验证集上！调参请在验证集上！调参请在验证集上！" / "保留一个真正不碰的最终测试集，最好按时间或业务单元切。重要版本才评一次。"
- **要点**: 验证集用于调参，测试集只能用于最终评估。反复在验证集上调参会使其成为训练的一部分（验证集过拟合）。最终测试集必须保持纯净。
- **参见**: [[cross-validation]], [[test-set-overfitting]]

### 验证集被反复调穿——保留最终测试集
- **来源**: BugBuster喵
- **适用场景**: 长时间迭代优化的项目
- **原文**: "很多人嘴上说没有用测试集，实际上每天看验证集调参数，调到最后验证集也成了训练的一部分。解决办法是保留一个真正不碰的最终测试集，最好按时间或业务单元切。重要版本才评一次。"
- **要点**: 连续迭代时会无意中让验证集信息影响调参决策。应冻结一个最终测试集，只在关键版本时评估一次。
- **参见**: [[train-val-test-split]]

### 观察loss胜于观察准确率
- **来源**: Captain Jack / Tang AI / 随机漫步的傻瓜
- **适用场景**: 训练过程中的监控决策
- **原文**: "准确率虽然是评测指标，但是训练过程中还是要注意loss的。你会发现有些情况下，准确率是突变的，原来一直是0，可能保持上千迭代，然后突然变1。要是因为这个你提前中断训练了，只有老天替你惋惜了。" / "loss，对于大模型训练仅能做一个基本的判断了，基本上，到了中后期，loss小，效果不一定好"
- **要点**: 准确率可能突变，loss的下降趋势更稳定可靠。但大模型训练中后期loss与效果解耦，需结合业务指标判断。
- **参见**: [[evaluation-metrics]], [[loss-function]]

### 不要只看loss，要看任务相关指标
- **来源**: Tang AI / anchor / BugBuster喵
- **适用场景**: 类别不平衡、排序、生成等任务
- **原文**: "对于负样本太多的数据集，测试集的loss降并不代表没有过拟合，试试看看f1或者auc有没有降低" / "不要只看精度，考虑其他metrics，并始终以上线后的可视化效果为最终评判标准。" / "训练loss下降很舒服，但业务指标可能不动。分类看校准，排序看NDCG、转化，检测看不同尺寸目标，生成看事实性和可用性。"
- **要点**: Loss下降可能只是因为负样本学得好。始终使用与业务目标对齐的评估指标（F1、AUC、NDCG、召回率、精确率等），不能只依赖loss。
- **参见**: [[evaluation-metrics]], [[class-imbalance]]

### 离线AUC提升不一定等于线上效果提升
- **来源**: BugBuster喵
- **适用场景**: 推荐系统、搜索排序、风控等线上A/B测试场景
- **原文**: "之前做一个电商搜索排序模型，离线AUC提了0.7个点，团队挺兴奋。后来线上灰度没动静，甚至部分品类略降。排查后发现验证集是随机切的……那一周我们调了学习率、特征交叉、负采样比例，基本都是在错误验证集上自嗨。"
- **要点**: 离线评估结果与线上效果不一致，常见原因包括验证集切分不当、评估指标与业务目标不匹配、线上数据分布偏移等。先确认评估可信度，再调参。
- **参见**: [[train-val-test-split]], [[evaluation-metrics]]

### 对比训练集和验证集的loss判断过拟合
- **来源**: Captain Jack / 无忧无虑
- **适用场景**: 训练过程中的诊断
- **原文**: "判断过拟合，训练是否足够，是否需要early stop的依据" / "模型在训练集上的误差很小……但是验证集的 loss 相对较大。那就是过拟合了。在训练集和验证集上的loss都比较大……那可能就是欠拟合了。"
- **要点**: 训练loss低+验证loss高=过拟合（加正则化、加数据）；两边loss都高=欠拟合（加模型容量、调学习率）。这是调参诊断的第一步。
- **参见**: [[overfitting]], [[underfitting]]

### 多任务学习中的loss平衡
- **来源**: Captain Jack
- **适用场景**: 多任务学习、多目标优化
- **原文**: "多任务情况下，各loss想法限制在一个量级上，或者最终限制在一个量级上，初期可以着重一个任务的loss"
- **要点**: 多任务的不同loss若在量级上差异过大，梯度会被大loss的任务主导。需要将各loss归一化到相近量级。
- **参见**: [[multi-task-learning]], [[loss-function]]

### 回归任务的输出也需要normalization
- **来源**: Captain Jack
- **适用场景**: 回归任务
- **原文**: "一般来说分类就是Softmax，回归就是L2的loss。但是要注意loss的错误范围（主要是回归），你预测一个label是10000的值，模型输出0，你算算这loss多大，这还是单变量的情况下。一般结果都是nan。所以不仅仅输入要做normalization，输出也要这么弄。"
- **要点**: 回归任务的输出值范围过大可能导致loss爆炸到NaN。输入和输出都需要做归一化处理。
- **参见**: [[data-preprocessing]]

### 数据预处理：zero-center与shuffle是基本功
- **来源**: BBuf / Captain Jack
- **适用场景**: CV任务的数据预处理
- **原文**: "数据与处理之zero-center。主要有2个步骤，第一个是减均值，第二个是除以方差。" / "shuffle, shuffle, shuffle."
- **要点**: zero-center（减均值除方差）使输入满足均值为0方差为1分布，一般减均值最常用。shuffle（数据打乱）对每个epoch必须执行，防止模型学到数据顺序相关的伪模式。
- **参见**: [[data-preprocessing]], [[../normalization/normalization]]

### 数据增强：CV用裁剪翻转旋转，NLP用同义词替换和回译
- **来源**: Tang AI / 无忧无虑 / 黑马程序员
- **适用场景**: 数据量不足或过拟合时的通用手段
- **原文**: "数据增强要结合任务本身来设计" / "图像裁剪，对称变换，旋转平移，都可以让模型在验证集上的表现更好。" / "数据越多越好！总是使用数据增强，如水平翻转，旋转，缩放裁剪等。"
- **要点**: CV领域：裁剪、翻转、旋转、色彩抖动、亮度变化、cutmix、mixup。NLP领域：同义词替换、随机丢弃词语、句子改写、回译（back-translation）。增强强度本身也是一个需要调节的超参数。
- **参见**: [[data-augmentation]], [[overfitting]]

### 类别不平衡：Focal Loss与类别加权
- **来源**: Tang AI / anchor / DOTA / 黑马程序员
- **适用场景**: 类别极度不平衡的分类任务（异常检测、长尾识别）
- **原文**: "Focal loss对于极大不平衡的数据集确实有奇效，其中gamma因子可以成10倍数衰减" / "如果你的分类精度不够是样本不均衡造成的，考虑使用focal loss。" / "如果你有高度不平衡的数据问题，在训练期间应用类别加权操作。"
- **要点**: Focal Loss通过调制系数降低易分类样本的loss权重，使训练更关注难样本。gamma通常从0.5开始尝试，以10倍步长搜索。也可用sklearn的class_weight计算类别权重。过采样/欠采样也是常用手段。
- **参见**: [[class-imbalance]], [[focal-loss]]

### Hard Negative Mining提升召回模型效果
- **来源**: Tang AI
- **适用场景**: 召回、对比学习、检索任务
- **原文**: "召回负采样负样本效果肯定高于曝光未点击负样本" / "粗排用精排后topk做负样本，基本是有效果的"
- **要点**: 召回任务的负样本选择非常关键。曝光未点击负样本优于随机负样本；用精排模型排名的top-K结果作为粗排的负样本是有效做法。
- **参见**: [[negative-sampling]], [[recommendation]]

### 数据也是超参数——采样比、增强强度、混合比例
- **来源**: BugBuster喵
- **适用场景**: 分类、检测、推荐、大模型SFT
- **原文**: "我越来越倾向于把数据也当成超参数。比如分类任务里，正负样本比例就是超参数。推荐任务里，负采样策略就是超参数。检测任务里，resize 尺寸和增强强度就是超参数。大模型 SFT 里，单轮和多轮数据比例就是超参数。"
- **要点**: 数据配置（采样比例、增强策略、混合比例、截断策略、模板格式）对效果的影响往往大于模型结构参数。应有意识地对数据参数进行调优。
- **参见**: [[hyperparameter-tuning]], [[data-augmentation]]

### 标签噪声检测与数据清洗
- **来源**: BugBuster喵
- **适用场景**: 任何有监督任务，尤其是标注质量不可控的场景
- **原文**: "标签噪声分析可以看 Cleanlab" / "我们重新清洗标签，按话术模板和时间切分，补了一批线上 hard case，模型参数几乎没大改，F1 在新验证集上从86提到89"
- **要点**: 标签噪声是模型效果差但不易察觉的常见原因。可使用Cleanlab等工具检测疑似错误标签。人工抽样验证标签一致性常常能发现30%的问题。
- **参见**: [[label-noise]], [[cleanlab]]

### Embedding可视化：先PCA降维再用t-SNE
- **来源**: Tang AI
- **适用场景**: 理解模型学到的表征、检查embedding质量
- **原文**: "对于embedding可视化的问题最好是先用pca降维，然后采用t-sne进行可视化，但是，对于类别很多的情况，个人建议还是肉眼抽样观察，t-sne只能对于类目不多的情况的embedding进行可视化。"
- **要点**: PCA用于初始降维（去噪声），t-SNE进一步降维到2D/3D可视化。t-SNE在类别多时效果不佳，此时应人工抽样观察bad case。
- **参见**: [[visualization]], [[embedding]]

### 数据版本管理（DVC）
- **来源**: BugBuster喵
- **适用场景**: 团队协作、实验可复现、数据追踪
- **原文**: "数据版本管理可以看 DVC"
- **要点**: 使用DVC（Data Version Control）对数据集进行版本管理，确保每次实验的数据版本可追溯、可复现。配合git使用。
- **参见**: [[experiment-management]], [[reproducibility]]

### 大模型评测：人工Eval Set + OpenCompass + RAGAS
- **来源**: BugBuster喵
- **适用场景**: LLM微调、RAG系统、大模型应用
- **原文**: "要做人工标注的 eval set，要覆盖典型问题、边界问题、长尾问题，还要记录失败类型。现在行业里稍微成熟一点的大模型团队，基本都会先搭评测，再谈微调。没有评测的大模型微调，就是在算力上投币。" / "大模型评测方面可以看 OpenCompass、HELM，中文场景再补一下 C-Eval。做 RAG 可以看看 RAGAS。"
- **要点**: 大模型时代不能只看loss或BLEU/ROUGE。需要构建人工标注的评估集，覆盖典型/边界/长尾问题。OpenCompass用于综合评测，RAGAS用于RAG系统评估。
- **参见**: [[llm-evaluation]], [[ragas]], [[opencompass]]

### 数据风格统一性决定模型是否学偏
- **来源**: Tang AI
- **适用场景**: 大模型SFT数据构建
- **原文**: "数据集的风格一定要统一，输出的答案风格一定要单一。比如，每次答案都是以json格式的，确保数据都是```json xxx```这种格式。如果sql的答案都是select a,b,c，那么就不要出现select *。保证你数据的统一性，纯洁性是模型不学偏的根本。"
- **要点**: SFT数据的输出风格必须高度一致，混入多种格式会导致模型输出不稳定。同一任务的数据模板、格式、答案风格应该严格统一。
- **参见**: [[sft-data]], [[prompt-format]]

### LLM模型越强，数据质量越重要
- **来源**: BugBuster喵
- **适用场景**: 大模型微调、数据构建
- **原文**: "这几年大模型很火，很多人有个误解，以为模型越强，数据就不重要了。实际正相反。模型越强，数据质量对最终效果的影响越明显。差数据会让强模型学得更快，也错得更自信。"
- **要点**: 强模型会更快地从数据中学习模式，如果数据存在噪声或不一致，错误也会被快速放大。数据质量是大模型微调的基石。
- **参见**: [[data-quality]], [[sft]]

### 数据分布偏移导致领域适配问题
- **来源**: 量子位（Josh Robin的debug流程）
- **适用场景**: 自动驾驶、跨领域迁移
- **原文**: "经过分析得出，训练集缺乏夜晚场景、反光等情况。后续将在训练集中加入此类数据纠正偏差。"
- **要点**: 训练集和测试集分布偏差的分析方法：使用两个验证数据集（一个来自训练集分布，一个来自测试集分布），对比各自的错误率来诊断分布偏移。
- **参见**: [[domain-adaptation]], [[data-distribution]]

### 错误分析比盲调参数更有价值
- **来源**: BugBuster喵
- **适用场景**: 调参陷入瓶颈时
- **原文**: "每轮重要实验都要看错误样例，整理失败类型。错误分析比盲调参数更有价值。" / "后来我们停下来做错误分析，发现漏掉的样本里，大量是新话术和变体表达，训练集里几乎没有。误杀样本则集中在几个正常业务词上……我们重新清洗标签，模型参数几乎没大改，F1在新验证集上从86提到89。"
- **要点**: 连续调参无提升时，停止调参转向错误分析。分类错误类型、检查数据分布、分析bad case，往往比继续调参收益更大。
- **参见**: [[error-analysis]], [[../debugging/debugging]]

### 随机种子必须固定，否则对比实验不可信
- **来源**: 爱睡觉的KKY / LindgeWAI / BugBuster喵
- **适用场景**: 所有实验对比场景
- **原文**: "随机数种子设定好，否则很多对比实验结论不一定准确" / "没有 seed，不记录环境，结果就很难复现。固定 seed 也不代表完全确定，但至少别让实验变成记忆游戏。"
- **要点**: 每次实验记录随机种子和运行环境。种子不固定时，小数据任务的结果方差可能大于调参带来的提升幅度。使用固定种子（如42、3407）进行对比实验。
- **参见**: [[reproducibility]], [[experiment-management]]

### 标签平滑（Label Smoothing）
- **来源**: DOTA
- **适用场景**: 分类任务的过拟合抑制
- **原文**: "Label smoothing将hard label转变成soft label，使网络优化更加平滑。它通常用于减少训练DNN的过拟合问题并进一步提高分类性能。"
- **要点**: 将hard label（0/1）转换为soft label（如0.9/0.1），防止模型对训练标签过分自信，提升泛化能力。实现：`targets = (1 - label_smooth) * targets + label_smooth / num_classes`。
- **参见**: [[../regularization/regularization]], [[overfitting]]

### 超参数间接依赖测试集导致的偏差
- **来源**: 白小鱼
- **适用场景**: 长期迭代的竞赛或研究项目
- **原文**: "机器学习方法的最终参数在训练时并不完全固定，而是间接依赖于测试集。……使用未校准的评价指标或使用测试集指导参数设定可能会产生误导性的结果，进而导致在实际环境下的差异性，这被称为是参数选择的偏差。"
- **要点**: 即使没有直接在测试集上训练，反复参照测试集结果调参也会引入选择偏差。最终线上部署时效果可能远低于预期。
- **参见**: [[test-set-overfitting]], [[evaluation-bias]]

### 确认分类网络学习充分——观察Softmax输出分布
- **来源**: Captain Jack
- **适用场景**: 分类任务的训练状态诊断
- **原文**: "如果是二分类，你会发现，刚开始的网络预测都是在0.5上下，很模糊。随着学习过程，网络预测会慢慢的移动到0,1这种极值附近。所以，如果你的网络预测分布靠中间，再学习学习。"
- **要点**: 分类网络的输出概率分布反映学习进度。如果预测概率集中在中间值（如0.4-0.6），说明网络尚未充分学习，需要继续训练。
- **参见**: [[classification]], [[model-convergence]]

### 训练集和验证集存在泄漏时的处理
- **来源**: BugBuster喵
- **适用场景**: 文本分类、风控、推荐系统
- **原文**: "再看数据切分，很多相似模板同时出现在训练和验证里，验证集高估了泛化。" / "按话术模板和时间切分，补了一批线上 hard case"
- **要点**: 当数据集存在模板结构或重复模式时（如自动生成的模板化文本），按模板ID或时间切分而非随机切分，才能真实评估泛化能力。
- **参见**: [[train-val-test-split]], [[data-leakage]]
