# 评估指标指南

## 是什么

评估指标（Evaluation Metrics）是衡量模型性能的定量标尺。同一个模型的"好"与"坏"完全取决于你选择的指标——用准确率评估不平衡分类任务会得到误导性的结论，用 BLEU 评估对话生成质量会遗漏语义相关性。评估指标本质上定义了"什么是好的预测"。

### 常见指标快速一览

| 指标 | 缩写 | 适用场景 | 核心关注点 |
|------|------|---------|-----------|
| 准确率 | Accuracy | 类别平衡的分类任务 | 整体正确率 |
| 精确率/召回率/F1 | Precision/Recall/F1 | 不平衡分类、信息检索 | 少数类性能 |
| AUC-ROC | AUC | 排序任务、不平衡分类 | 正负类排序质量 |
| AUC-PR | PR-AUC | 极端不平衡分类 | 正类精确率-召回率权衡 |
| 交叉熵损失 | Cross-Entropy Loss | 训练过程监控、概率校准 | 预测概率的质量 |
| NDCG | NDCG | 搜索排序、推荐系统 | 排序列表中位置加权质量 |
| Perplexity | PPL | 语言模型 | 模型对序列的预测能力 |
| BLEU/ROUGE | BLEU/ROUGE | 机器翻译、摘要 | n-gram 重叠度 |

## 为什么重要

### 指标驱动优化方向

模型调参的每一个决策——改学习率、加正则化、调数据比例——都需要一个信号来回答"我做得更好了吗"。如果这个信号是错误的，所有"提升"都是假提升。这就是为什么[[../data-eval/data-eval|先确认评估可信度，再调参]]是第一原则。

### 不同指标对同一模型给出不同结论

一个在 99:1 不平衡数据上训练的模型：准确率 = 99%（全部预测为多数类），F1 = 0（没有预测出任何一个少数类）。用准确率调参会得到"最好的模型是全部分类为多数类"这个完全无用的结论。因此，**指标选择本身就是模型设计的一部分**。

### 离线与在线指标的分裂

离线 AUC 提升 0.7 个点，线上 A/B 测试可能完全没有变化，甚至更差。常见原因包括：
- 验证集切分不当（[[./cross-validation|时序数据随机切分导致的信息泄漏]]）
- 离线优化指标与线上业务目标不匹配（如离线优化 AUC，线上关注转化率）
- 数据分布漂移（训练/验证集分布与线上真实分布不一致）

### Loss 对大模型和小模型的不同意义

对小模型（如标准 CNN、ResNet），训练 loss 的下降趋势是可靠的收敛信号，loss 与业务指标相关性高。

对 LLM，训练中后期的 loss 下降与业务效果之间出现解耦：loss 继续降低但生成质量不再提升，甚至可能因为过拟合训练数据分布而导致通用能力下降。因此仅靠 Loss 评估 LLM 是不充分的，必须配合[[../data-eval/data-eval|人工标注 Eval Set + OpenCompass + RAGAS]]等多元化评测。

## 如何使用

### 按任务类型选指标

**分类任务**：
- 类别平衡 → Accuracy。
- 类别不平衡 → 首选 F1（宏平均 F1 或加权 F1）和 AUC-PR。当正类极稀有时，AUC-ROC 可能过于乐观（ROC 对不平衡不敏感），AUC-PR 更反映实际。
- 需要概率校准 → 加上 Log Loss（交叉熵），关注预测置信度是否合理。

**排序/推荐任务**：
- NDCG 和 MAP（Mean Average Precision）是最常用的排序指标。NDCG 能体现排序位置的重要性——排在前面的文档贡献更大。
- AUC 作为辅助指标，但 AUC 对所有位置一视同仁，不反映排序的业务价值。

**生成/LLM 任务**：
- 核心指标：人工评分、胜率（Win Rate）、有用性评分。
- 辅助自动指标：在翻译/摘要等格式固定任务上 BLEU/ROUGE 可作为参考；在开放生成中，G-Eval、BLEURT 等基于模型的自动评估更接近人工判断。
- 绝不单看 Loss：[[../data-eval/data-eval|Loss 对大模型训练仅能做基本判断]]。

### 实践原则

**分层评估体系**：
1. 监控 Loss（训练诊断）
2. 计算核心指标（模型选择）
3. 人工检查 Bad Case（发现隐性问题）
4. 在线 A/B 测试（最终验证）

每层回答不同的问题：Loss 回答"是否在学"；核心指标回答"学得好不好"；Bad Case 回答"哪里没学好"；在线测试回答"真的有用吗"。

### 多指标冲突时的决策

不同指标给出矛盾信号时（如 Loss 下降但 F1 不动），优先相信**与业务目标最接近的指标**。如果业务目标是"尽可能少漏掉负类"，那就以召回率和 F1 为准，而不是 Loss。

## 常见误区

### 误区一：只看 Loss

这是最常见的错误。Loss 下降只代表模型对训练数据的拟合度在提升，不代表泛化能力或业务效果在提升。尤其在大模型场景中，中后期 Loss 与效果解耦——需要结合 [[./class-imbalance|任务相关指标]] 来判断。

### 误区二：离线指标等于线上效果

离线 AUC 提升 1% 不代表线上转化率会提升 1%。离线评估无法完全模拟线上的数据分布、延迟约束、用户行为反馈循环。离线指标更多是"**门禁**"而非"**目标**"——不通过一定不部署，通过了也不一定有效。

### 误区三：重复利用测试集调参

在同一个测试集上反复评估，测试集信息会无形中影响调参决策，最终测试集失去评估泛化能力的作用。必须[[./cross-validation|保留一个真正不碰的最终测试集]]，只有发布前才评估一次。

### 误区四：多任务用单一指标衡量

多任务学习、多目标优化中，不同任务可能有不同的最优指标。加权合并单一数字做法虽然方便，但掩盖了任务的独立表现。应分别报告每个任务的指标，并附上加权总分的具体权重。

### 误区五：忽略置信度校准

高准确率但置信度偏差大（如预测概率 0.9 但实际正确率只有 0.7）的模型在实际部署中可能造成问题——尤其是在风控、医疗等需要可信概率估计的场景。应使用预期校准误差（ECE）评估校准质量。
