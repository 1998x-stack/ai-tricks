# 多任务学习

## 是什么

多任务学习（Multi-Task Learning, MTL）在推荐系统中指用一个共享模型同时预测多个目标——如同时预估点击率（CTR）和转化率（CVR），或同时预测点赞、收藏、转发等互动行为。典型结构是 Shared-Bottom 共享底层 + 多个 Task Tower，每个 Tower 负责一个子任务。

推荐系统的关键特点是**多个目标之间存在复杂关系**：点击是转化的前提但不等同，点赞和收藏有不同的用户意图。如何平衡这些任务的学习是 MTL 的核心挑战。

## 为什么重要

用单独的模型分别预测每个目标，会带来三个问题：

- **计算资源浪费**：维护多个独立模型意味着更多的训练、存储、推理开销。
- **样本量瓶颈**：转化、购买等深层目标的正样本远少于点击，单任务模型的数据利用率不足。
- **忽略任务间关联**：点击和转化不是独立的——先有点击才有转化。MTL 通过共享表征让浅层目标（点击）帮助深层目标（转化）学习。

但在实践中，MTL 的挑战也非常现实：不同任务的 Loss 量级差异巨大、某一任务主导梯度导致其他任务"学不动"、任务间存在冲突（如点击率高但转化率低的"标题党"内容）。

## 如何使用

### Gradient Surgery（推荐方案）

Gradient Surgery 是解决多任务梯度冲突的最实用方法。其核心思想是：当两个任务的梯度方向冲突（夹角 > 90 度）时，将一个任务的梯度投影到另一个任务梯度的正交方向上，消除冲突分量。

具体做法（PCGrad 算法）：
1. 采样一个 batch，分别计算每个任务的梯度。
2. 对所有任务对 (i, j)，如果 grad_i 与 grad_j 的夹角大于 90 度，将 grad_i 中与 grad_j 冲突的分量移除：`grad_i = grad_i - proj(grad_i, grad_j)`。
3. 使用修正后的梯度更新参数。

Gradient Surgery 的效果稳定、实现简单（几十行代码即可完成），且不需要引入额外超参数。

### 不推荐 Pareto 优化

Pareto 优化（多目标同时达到 Pareto 最优）理论上是 MTL 的最优解，但实践中：

- 计算复杂度高：需要求解多目标优化的 Pareto 驻点。
- 效果提升有限：Tang AI 的经验表明"帕累托就算复杂度又高，效果也没有好很多"。
- 调参困难：多个目标之间的权重调节本身就是一个难题。

**建议：先用 Gradient Surgery 快速拿到可用的多任务模型，除非业务对多个目标的均衡有极其严格的要求，否则不必上 Pareto 优化。**

### 各 Loss 必须在同一量级

不同任务的 Loss 量级差异过大会导致训练被大 Loss 任务主导。例如：
- CTR 预测用 BCE Loss，值域约 [0, 0.7]。
- CVR 预测用 BCE Loss，值域约 [0, 0.1]。
- 如果 CTR 的 Loss 平均 0.3 而 CVR 平均 0.05，梯度会被 CTR 主导，CVR 任务"学不动"。

解决方法：
- 对每个任务的 Loss 做缩放，使它们在量级上接近。例如各 Loss 乘以权重系数 w_i，使得 w_i * Loss_i 约等于同一量级。
- 初期可以着重某一主任务的 Loss，待主任务收敛后再逐步平衡次要任务。
- 也可以使用 Uncertainty Weight 方法（Homoscedastic Uncertainty），让模型自适应学习各任务的权重。

### SENet / LHUC：可观察特征重要性

SENet（Squeeze-and-Excitation Network）和 LHUC（Learning Hidden Unit Contribution）是推荐系统中**为数不多的可以直接观察特征重要性的模型结构**，特别适合用来分析多任务场景中各特征对不同任务的贡献。

**SENet**：
- Squeeze：对特征做全局平均池化，压缩成一个描述子。
- Excitation：通过两个 FC 层 + Sigmoid 学习每个特征的"重要性权重"。
- Scale：将权重乘回原始特征，增强重要特征、抑制不重要特征。
- 在推荐系统中，SENet 输出的权重可以直接解释为"这个特征在当前任务中的重要程度"。

**LHUC**：
- 为每个任务学习一个隐层单元的缩放系数。
- LHUC 权重可以直观反映某个隐层神经元对特定任务的贡献。
- 相比 SENet 在特征层面做权重，LHUC 在神经元层面做权重，更适合多任务 Tower 的分析。

两者可以组合使用：SENet 做特征筛选，LHUC 做任务特定适配。

## 常见误区

- **所有 Loss 不加权重直接相加**：不同任务 Loss 量级差异可以到 10 倍以上，直接相加等于废弃了部分任务。
- **一上来就用复杂 MTL 结构**：推荐的做法是先用 Shared-Bottom + Gradient Surgery 作为 baseline，确认存在 gradient conflict 之后，再考虑 MMoE、PLE 等复杂结构。
- **把 Gradient Surgery 当万能药**：Gradient Surgery 解决的是梯度冲突问题，但解决不了"数据量不足以支撑多任务学习"或"标签质量差"等根本问题。
- **忽略任务间的样本依赖**：CVR 依赖 CTR（先点击才可能转化），在数据构造时必须考虑这种漏斗关系。直接在所有样本上训练 CVR 模型会引入 Selection Bias。
- **MTL 总比单任务好**：如果多个任务之间几乎没有共享信息（完全不相关的预测目标），强制共享底层网络反而会导致"负迁移"，效果不如各自独立训练。
